## 0.机器人本质
我们用**第一性原理**来拆开看这件事：

你看到的流程是：

```
策略 (policy)
↓
控制器集成
↓
机器人通信
↓
真实机器人运行
```

这其实是在解决一个最根本的问题：

> **如何把一个数学函数（policy）变成真实世界机器人的物理动作。**

强化学习最后得到的只是一个**函数**：

$a = \pi(s)$  

意思是：

```
输入：机器人状态 s
输出：控制动作 a
```

但真实机器人不是函数，它是一个**物理系统**。  
所以必须有一整套系统把这个函数**嵌入真实机器人的控制循环**。

下面一步一步用第一性原理拆开。

---

### 一、第一性原理：机器人控制的本质是什么

机器人在物理世界里，本质是：

```
传感 → 计算 → 执行
```

循环：

```
state_t  → controller → action_t
```

也就是：

```
观测
↓
决策
↓
执行
```

这个循环通常是 **200Hz~1000Hz**。

例如：

```
每 5ms
```

系统要完成：

```
读取传感器
运行控制算法
发送电机命令
```

---

### 二、强化学习策略只是“决策函数”

强化学习训练完得到：

```
policy.pt
```

它只是一个神经网络。

本质是：

```
f(x) = y
```

输入：

```
机器人状态
```

输出：

```
关节命令
```

但问题是：

**机器人本身不会运行 PyTorch。**

所以需要把它嵌入机器人控制系统。

---

### 三、控制器集成（Controller Integration）

这是第一步。

目的：

> **把神经网络策略嵌入机器人控制循环**

机器人控制器通常是 C++ 写的。

例如：

```
while(true):

    state = readSensors()

    action = policy(state)

    sendMotorCommand(action)
```

这里的关键是：

#### 1 读取状态

机器人传感器：

```
IMU
joint position
joint velocity
foot contact
```

这些组合成：

```
observation
```

---

#### 2 运行策略

策略通常是：

```
MLP
```

例如：

```
obs → 512 → 256 → action
```

运行一次前向：

```
action = network(obs)
```

---

#### 3 输出控制命令

例如：

```
desired joint position
desired torque
desired velocity
```

然后发给电机控制器。

---

### 四、机器人通信（Robot Communication）

第二步。

机器人系统不是一个单进程程序。

它通常是一个**分布式系统**。

结构类似：

```
计算机
↓
网络
↓
机器人主控
↓
电机驱动
```

所以需要通信。

例如：

```
PC → robot
```

发送：

```
joint commands
```

同时接收：

```
sensor data
```

---

Unitree 使用的是：

```
DDS (Data Distribution Service)
```

这是一种实时机器人通信协议。

类似：

```
ROS2
```

但更底层。

---

通信结构是：

```
policy process
↓
DDS
↓
robot controller
↓
motor drivers
```

---

### 五、运行在真实系统

第三步。

这一步最关键。

仿真里的控制：

```
policy → simulator
```

但真实机器人：

```
policy
↓
controller
↓
communication
↓
robot hardware
↓
physics
```

差异很大。

因为真实系统存在：

```
延迟
噪声
摩擦
模型误差
```

例如：

```
policy计算 3ms
通信延迟 5ms
电机响应 8ms
```

总延迟：

```
16ms
```

而仿真可能是：

```
0ms
```

这就是 **sim2real gap**。

### 六、为什么必须有这一整套 pipeline

如果没有：

```
controller integration
communication
deployment
```

那么 RL 代码只能：

```
在仿真跑
```

永远不能：

```
控制机器人
```

所以机器人公司最重要的工程之一就是：

> **把策略嵌入真实控制系统**

### 八、最核心的一句话

控制器集成 + 机器人通信 + 真实系统运行，本质是在解决：

> **如何把一个神经网络决策函数嵌入到一个实时控制的物理系统中。**

## 1.Actuator 和 Controller 的区别

### 一、Actuator 和 Controller 的区别

先给你一句**最核心的区别**：

> **Controller 决定“应该怎么动”，Actuator 负责“真的让它动起来”。**

**policy.onnx**
**↓**
**C++ controller**
**↓**
**DDS / robot SDK**
**↓**
**真实机器人**

---

#### 1 Controller（控制器）

控制器是一个**算法 / 计算模块**。

作用是：

```
输入：当前状态
输出：控制指令
```

例如：

```
机器人状态：
关节角度
关节速度
IMU

↓

控制器计算

↓

关节目标位置
```

例如：

```
policy(state) → action
```

强化学习里的 **policy 本质就是 controller**。

例如：

```
RL policy
MPC
PID controller
```

这些都是 controller。

---

#### 2 Actuator（执行器）

Actuator 是**物理设备或其模型**。

作用：

```
把控制信号 → 变成机械运动
```

例如：

```
控制器输出：

target_position = 0.5

↓

电机驱动

↓

关节真的转到 0.5
```

所以 actuator 是：

```
电机
减速器
驱动器
```

组合。

---

#### 3 控制系统的完整结构

机器人系统其实是：

```
Controller
    ↓
Control Signal
    ↓
Actuator
    ↓
Robot Dynamics
```

例如：

```
RL policy
↓
target joint position
↓
motor
↓
joint movement
```

---

### 二、IdealPDActuator、DelayedPDActuator、DCMotor、ImplicitActuator 、UnitreeActuator到底有什么区别？

1）ImplicitActuator 
“隐式”隐掉的不是 `P` 和 `D`，而是：

**把“驱动力怎么进入下一时刻状态”这件事，放进了求解器联立求解，而不是先显式算完再积分。**

第一性原理地说，系统在做两件事：

1. 根据当前位置/速度误差产生驱动力
2. 用这个驱动力更新下一时刻状态

显式 PD 和隐式 PD 的核心区别就在第 2 步怎么做。

**显式 PD**
思路是：

1. 用当前时刻状态 \(q_t, \dot q_t\) 算 torque
$$
\tau_t = K_p(q^* - q_t) + K_d(\dot q^* - \dot q_t)
$$

2. 再用这个 torque 去推进系统到下一步
$$
q_{t+1}, \dot q_{t+1}
$$

也就是说：
- 力是先算出来的
- 更新状态是后做的
- 这是“先算控制，再做积分”

这里的问题是，当 \(K_p\) 很大、接触很强、步长不够小时，系统会很“硬”，数值上容易炸。

**隐式 PD**
思路是：

它不先固定用 \(q_t, \dot q_t\) 算一个 torque。  
而是把“PD 想满足的约束”和“动力学方程”“接触约束”一起解。

直觉上它更像在求：

$$
\text{求 } q_{t+1}, \dot q_{t+1}
$$
使得：
- 动力学成立
- 接触约束成立
- 关节驱动倾向于靠近目标 $q^*, \dot q^*$

所以驱动力不是先单独外算死，而是在“下一步状态未知”的情况下，和下一步状态一起被联立确定。

这就是“隐式”：
- **显式**：$\tau_t$ 显式写成当前状态的函数
- **隐式**：$\tau$ 通过求解器对下一步状态的联立求解隐含地产生

如果用离散时间视角看：

显式更像：
$$
\tau_t = K_p(q^*-q_t) + K_d(\dot q^*-\dot q_t)
$$
$$
x_{t+1} = f(x_t, \tau_t)
$$

隐式更像：
$$
x_{t+1} = f(x_t, \tau(x_{t+1}))
$$

 $$ \tau(x_{t+1}) \approx K_p(q^* - q_{t+1}) + K_d(\dot{q}^* - \dot{q}_{t+1}) $$ 这里不一定是求解器真实源码里的精确写法，但它表达了关键点： 
 - $K\_p$ / $K\_d$ 仍然在驱动力模型里 
 - 区别只是误差不再只看当前时刻，而是和下一步解联立

这就是它稳定的根源：
- 它不是拿当前误差猛打一拳
- 而是把驱动目标作为下一步联立解的一部分
- 所以对大刚度、强接触更稳

你问“隐了什么”，最本质的答案就是：

**隐掉的是对 actuator force / torque 的显式前向计算过程。**

你在代码外面看不到：

$$
\tau = K_p e + K_d \dot e
$$

这条力矩先被算出来再送进系统的过程。  
你只提供：
- target
- stiffness
- damping

剩下的由求解器和动力学、约束一起解。

所以和显式 PD 的第一性区别，不是“有没有 P 和 D”，而是：

- **显式 PD**：控制律先独立算出输入，再积分系统
- **隐式 PD**：控制律作为离散动力学方程的一部分，被一起求解

一句话压缩版：

**隐式 PD 隐掉的是“先算 torque 再积分”这个显式控制输入步骤；它把 PD 驱动并入了下一时刻状态的联立求解。**

2）IdealPDActuator
先按 **PD+前馈公式来算扭矩，再做一个与速度无关的常数上限裁剪**；它不会像 DCMotor 那样根据当前关节速度去按扭矩-速度曲线动态收缩可用扭矩范围。 
  $$\tau_{computed} = k_p(q_{des} - q) + k_d(\dot{q}_{des} - \dot{q}) + \tau_{ff}$$  
然后再做一次 $$\tau_{applied} = clip(\tau_{computed}, -\tau_{max}, \tau_{max})$$这里的 $\tau_{max}$ 是一个固定的最大扭矩上限，来自配置里的最大 effort / 电机能力，不随当前速度变化。也就是说，它有静态扭矩限幅，但没有 `torque-speed relation` 那种速度相关裁剪。 
  
3）DCMotor
先用和 IdealPDActuator 一样的公式算 $\tau_{computed}$，再根据当前关节速度，用线性的四象限 **DC motor torque-speed curve** 算出此刻允许的最小/最大扭矩，再裁剪。
所以两者最大的差别就在第二步：
- IdealPDActuator = PD + 前馈 + 固定扭矩裁剪 
- DCMotor = PD + 前馈 + 速度相关扭矩裁剪

4）DelayedPDActuator
这个类的本质很简单，**它不是电机扭矩-速度模型，而是“有延迟的理想 PD”**，它继承自 IdealPDActuator，先把控制指令延迟若干个 physics step，再按 IdealPDActuator 的方式去算扭矩。
它延迟的是“命令”，不是“机器人状态”。当前时刻的 `joint_pos` 和 `joint_vel` 还是机器人现在的状态；被延迟的是发给 actuator 的目标位置、目标速度、前馈力矩这三类控制输入。
用一个时间线例子看就很直观 ：
假设 `delay = 2` 个 physics step。 
你在时刻： 
- step 10 发命令：`q_des = 0.3` 
- step 11 发命令：`q_des = 0.5` 
- step 12 发命令：`q_des = 0.8` 
那么在 step 12 真正参与 PD 计算的，不是刚发的 `0.8`，而是 step 10 的 `0.3`。 也就是机器人当前状态已经走到 step 12 了，但 actuator 还在拿“两步前的命令”工作。这个“lagged by a certain number of physics steps”正是官方对它的定义。

5）UnitreeActuator
是Unitree RL Lab仓库自己实现的 UnitreeActuator。它： 
- 继承自 `DelayedPDActuator` 
- **也就是说它先保留了命令延迟** 
- **然后额外实现了一个 Unitree 风格的扭矩-速度曲线** 
- **再额外减去一个摩擦项** 
这个自定义曲线用了几组参数： 
- `X1`：满扭矩下的最高速度（knee point） 
- `X2`：空载速度 
- `Y1`：扭矩与速度同向时的峰值扭矩 
- `Y2`：扭矩与速度反向时的峰值扭矩 
- `Fs` / `Fd` / `Va`：静摩擦、动摩擦和摩擦激活速度。 
代码里 `UnitreeActuator` 的裁剪逻辑是： 
- 先判断当前输出扭矩和关节速度是不是同向 - 同向时用 `Y1`，反向时用 `Y2` 
- 当速度超过 `X1` 后，允许扭矩从该上限线性下降到 `X2` 附近的 0 
- 最后再叠加摩擦修正。 
对 Go2，当前默认 `UnitreeActuatorCfg_Go2HV` 给的是： 
- `X1 = 13.5`
- `X2 = 30` 
- `Y1 = 20.2` 
- `Y2 = 23.4`，并配合 `stiffness=25.0`、`damping=0.5`、`friction=0.01` 在 `UNITREE_GO2_CFG` 中使用。
