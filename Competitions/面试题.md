# BAAI 大脑大模型赛道面试题

> 相关材料：[[11-17智源BAAI大脑大模型赛道决赛答辩-浅试团队.pdf]]

## 1. 怎么给数据打分，依据是什么，Prompt 是什么？

### 面试主回答（约 1 分钟）

我们定义的 1–6 分不是原始数据集的“错误率”，而是转换后样本的**训练可用性等级**。虽然 Visual Genome、LVIS、Ego4D、EPIC-KITCHENS 等公开数据集本身有比较规范的标注流程，但我们还会把原始标注转换为空间问答、结构化 CoT 或多步规划数据。转换过程中可能出现视觉证据不足、空间关系反转、动作片段与文本错位、计划不可执行、输出格式错误等新问题，因此需要对最终的 `instruction + input + output` 再做任务相关的质量评估。

整个流程分为五层：

1. **硬规则校验**：检查图像/视频是否存在，字段是否完整，输出能否解析，坐标是否越界，是否为空输出或包含非法格式。
2. **相关性过滤**：计算 `instruction + input` 与 `output` 的语义相关性，过滤明显答非所问的样本。语义相似度只作为低成本闸门，不能判断视觉事实是否正确。
3. **奖励模型粗排**：使用 Skywork-VL Reward 对图像、问题和回答给出连续 reward，用于批量排序和筛掉明显低质量样本。
4. **强 VLM 复核与 1–6 分档**：使用 Qwen3-VL-Plus 按统一 rubric 检查视觉一致性、问答相关性、计划可执行性、格式合规性和训练价值，并输出 1–6 分、问题原因及保留建议。
5. **去重与分桶采样**：使用 SimHash、MinHash 和语义向量去重，再按空间理解、动态理解、任务规划等能力桶采样，形成约 20k 的 MU-Data，而不是简单选 reward 最高的 20k 条。

需要明确：**Skywork-VL Reward 的连续 reward 和 1–6 分不是同一个分数**。前者适合大规模相对排序；后者是可解释的离散质量等级，方便设定保留、修复和删除策略。

### 1–6 分的依据

评分优先级是：**事实正确性 > 任务相关性 > 推理/计划可执行性 > 格式合规性 > 训练价值**。不建议在面试中编造未实际使用过的精确加权公式，应将其解释为有明确锚点的等级评分。

| 分数 | 判定标准 | 处理方式 |
| --- | --- | --- |
| 1 | 媒体缺失、字段损坏，或答案与视觉/源标注明显矛盾 | 删除 |
| 2 | 与任务相关，但核心物体、空间关系、动作或答案错误 | 删除 |
| 3 | 基本相关，但信息缺失、歧义明显、CoT 不自洽或计划不可执行 | 修复或边界抽检 |
| 4 | 内容正确可用，仅有轻微冗余、表达或格式问题 | 按能力桶保留 |
| 5 | 视觉有据、问答匹配、计划可执行、格式完全合规 | 保留 |
| 6 | 满足 5 分，并覆盖长尾类别、复杂空间关系或高价值规划能力 | 优先保留 |

6 分不代表答案更长，而代表证据更完整、信息密度更高、对目标能力更有训练价值。

### 真实数据通常长什么样？

下面区分三种内容：

- **官方原始记录**：来自数据集官方文档或 Dataset Viewer，可作为面试中的真实例子。
- **字段结构裁剪**：保留真实字段和语义，用 `...` 省略大数组、图像二进制和无关元数据。
- **训练转换示意**：展示我们如何把原始标注转成 `instruction-input-output`，不是声称数据集原文件就长这样。

#### 1. Visual Genome：图片 + 区域描述 + 对象关系

```text
[图片占位：一名男子和一名女子坐在河边长椅上的场景]
```

官方论文展示的一个真实场景描述及关系为：

```json
{
  "region_description": "a man and a woman sit on a park bench along a river",
  "objects": ["man", "woman", "bench", "river"],
  "relationships": [
    ["man", "sits on", "bench"],
    ["man", "in front of", "river"],
    ["woman", "sits on", "bench"]
  ]
}
```

Visual Genome 原始下载文件还会包含 `image_id`、`object_id`、bounding box、relationship id 和 WordNet synset 等字段。上面是为了面试展示而裁剪后的真实语义，不是完整原始 JSON。

可能的训练转换示意：

```json
{
  "instruction": "根据图像判断男子与河流的空间关系。",
  "input": "<image>",
  "output": {
    "evidence": ["man", "river"],
    "relation": "in front of"
  }
}
```

评分时应检查 `relation` 是否仍由图片或原始关系三元组支持，而不能只检查问答文本是否相似。

#### 2. EPIC-KITCHENS：视频片段 + narration + 时间范围

```text
[图片占位：第一人称厨房视频中，手正在打开柜门；可并排放起始帧和结束帧]
```

官方注释仓库给出的真实 CSV 行字段示例：

```csv
narration_id,video_id,start_timestamp,stop_timestamp,start_frame,stop_frame,narration,verb,noun
P01_01_0,P01_01,00:00:00.14,00:00:03.37,8,202,open door,open,door
```

转换成规划训练数据后可能类似：

```json
{
  "instruction": "根据视频给出完成开门动作的高层步骤。",
  "input": {
    "video": "<frames 8-202>",
    "narration": "open door"
  },
  "output": {
    "plan": ["定位门把手", "抓住门把手", "拉开门"]
  }
}
```

这里 narration 和时间范围是官方标注，而三步 plan 是派生内容。派生步骤是否真的可从视频中得到，需要重新审核。

#### 3. BridgeData V2：多帧观测 + 7 维动作 + 语言指令

```text
[图片占位：WidowX 机械臂将胡萝卜放到盘子上的连续三帧：初始、抓取、完成]
```

BridgeData V2 采用轨迹/step 结构。下面保留真实字段形态，用 `...` 省略图像和数值数组：

```yaml
episode:
  steps:
    - observation:
        image_0: "<RGB image bytes>"
        image_1: "<RGB image bytes>"
        state: [...]
      action: [dx, dy, dz, droll, dpitch, dyaw, gripper]
      language_instruction: "put the carrot on the plate"
      is_first: true
      is_last: false
    - observation: { image_0: "<next frame>", state: [...] }
      action: [...]
      language_instruction: "put the carrot on the plate"
      is_last: false
```

单条轨迹不仅是“图片 + 一句话”，还包含按时间排列的观测和连续动作。映射到 AI2-THOR 规划模板时，要检查对象、动作空间和物理条件是否仍然成立。

#### 4. PixMo-Points：图片 URL + 哈希 + 指代表达 + 点坐标

```text
[图片占位：一个带前袋的背包或服装图片，在 front pocket 中心绘制一个青色点]
```

官方 Dataset Viewer 中可见的真实样本字段之一：

```json
{
  "image_url": "<official viewer image URL>",
  "image_sha256": "71dedeab9da3dd5a888101cc5cff923e4fb461e32ab35e8a5458ea1fba85d6b8",
  "label": "front pocket",
  "points": [
    {"x": 60.01943588256836, "y": 35.11602020263672}
  ]
}
```

另一个官方样本的 `label="elbow"`、`points=[]`，表示目标在图中不存在或没有有效指向点。审核时除了判断 label 是否相关，还应检查图片哈希、点是否越界、目标是否可见，以及空点是否代表合法负样本。

#### 5. AI2-THOR：动作请求 + Event 环境真值

```text
[图片占位：AI2-THOR 厨房场景，画面中标出 Apple 和可交互位置]
```

动作调用和返回信息的真实字段形态：

```python
event = controller.step(
    action="GetObjectInFrame",
    x=0.64,
    y=0.40,
    checkVisible=False
)

object_id = event.metadata["actionReturn"]
success = event.metadata["lastActionSuccess"]
```

对象 metadata 还包含：

```json
{
  "objectId": "Apple|-1.0|+1.0|+1.5",
  "objectType": "Apple",
  "visible": true,
  "position": {"x": -1.0, "y": 1.0, "z": 1.5},
  "pickupable": true,
  "isPickedUp": false,
  "parentReceptacles": [...]
}
```

因此 AI2-THOR 计划是否成功，应优先由 `lastActionSuccess`、对象状态和可达位置等环境真值判断，而不是只让 VLM 判断计划“听起来是否合理”。

#### 6. 一条完整的待评分样本长什么样？

```text
[图片占位：与 instruction 对应的图片，或从视频中均匀抽取的 4-8 帧]
```

送入质量审核器的数据可以统一整理为：

```json
{
  "dataset": "EPIC-KITCHENS",
  "task_type": "high_level_planning",
  "media": ["<image/frame paths>"],
  "source_annotation": {
    "narration": "open door",
    "start_frame": 8,
    "stop_frame": 202
  },
  "instruction": "根据视频给出完成开门动作的高层步骤。",
  "input": "<video frames>",
  "output": {
    "plan": ["定位门把手", "抓住门把手", "拉开门"]
  }
}
```

Judge 的核心任务是区分哪些字段来自官方标注，哪些字段是我们的转换结果，并重点审核转换结果有没有超出证据范围。

### 用真实数据片段解释为什么要评分

#### Visual Genome：原关系正确，不代表转换后的 CoT 正确

Visual Genome 的典型场景图包含类似：

```text
region: a man and a woman sit on a park bench along a river
relationships:
  sits_on(man, bench)
  in_front_of(man, river)
  sits_on(woman, bench)
```

如果转换后的回答写成“男人在河流后方”，问题来自我们的转换链路，而不是 Visual Genome 原始数据。审核时需要将输出重新对齐到关系三元组或图像证据。

#### EPIC-KITCHENS：动作标注不等于完整计划

官方动作标注示例包含：

```text
narration = "open door"
start_frame = 8
stop_frame = 202
verb = "open"
noun = "door"
```

这些字段能说明某段视频中发生了开门动作，但不能直接证明模型扩展出的“接近门、定位把手、抓住把手、拉开门”等每一步都在视频中出现。因此，从动作片段生成多步计划后，需要检查新增步骤是否有证据、顺序是否合理。

#### BridgeData V2：真实轨迹成立，不代表跨环境映射成立

官方示例指令包括：

```text
put the carrot on the plate
flip the pot upright
put the mushroom in the pot
move the cloth to the left
```

这些指令在 BridgeData 的机器人和场景中有对应轨迹，但映射到 AI2-THOR 后，场景可能不存在对应对象、可达位置或动作接口。因此跨数据集映射必须单独验证，不能仅凭语言流畅度判定为高质量。

#### AI2-THOR：优先用环境真值验证计划

AI2-THOR 的 Event metadata 提供 `lastActionSuccess`、`actionReturn`、对象状态、位置、可达点等字段。对 DPO 的 chosen 计划，应优先依据成功 episode 和环境状态验证；rejected 计划即使由脚本注入错误，也要确保错误真实影响可执行性，而不是只做无意义的同义改写。

#### ShareRobot：模板生成规模大，仍需防止重复和模板捷径

ShareRobot 将 51,403 个 planning episode 通过多类问题模板扩展为 1,027,990 个 QA，并由标注员监督生成过程。原始 episode 可以是高质量的，但模板扩展后仍可能出现高度重复、答案位置泄漏、问题表述与场景不匹配，因此需要模板级去重和派生 QA 复核。

### 推荐的 Judge Prompt

```text
你是一名具身多模态训练数据审核员。你将收到图像/视频、源标注，
以及转换后的 instruction、input 和 output。请评估该样本是否适合训练。

审核原则：
1. 只使用媒体内容和源标注，不补充无法验证的事实。
2. 语义相关不代表事实正确；如果输出与源标注冲突，以可验证证据为准。
3. 空间任务检查目标物体、指代、空间关系和坐标是否一致。
4. 视频任务检查动作是否出现在指定时间段，动作顺序是否正确。
5. 规划任务检查对象是否存在、前置条件是否满足、步骤顺序是否合理且可执行。
6. 检查输出是否满足指定 JSON schema；不要因回答更长而提高分数。
7. 如果源标注不足以支持派生结论，将其标记为 unsupported，不要自行推断。

评分维度：
- visual_grounding：视觉内容与源标注的一致性
- qa_relevance：instruction/input 与 output 的相关性
- reasoning_executability：CoT 自洽性或计划可执行性
- format_compliance：字段、JSON 和任务输出格式是否合规
- training_value：是否覆盖有效、困难或长尾能力，而非重复模板

请输出严格 JSON：
{
  "score": 1到6的整数,
  "hard_fail": true或false,
  "evidence": ["支持判断的视觉或源标注证据"],
  "issues": ["具体问题及对应字段"],
  "decision": "keep、repair或drop"
}
```

### 需要主动承认的限制

这套评分不是绝对真值。奖励模型和 VLM Judge 都可能有偏置，尤其不擅长精确坐标、多视角几何和长链规划。答辩结果也体现了这一点：EmbSpatial、MMSI、RealWorldQA 等与结构化推理和偏好对齐一致的指标提升明显，但 OmniSpatial、RefSpatial、Where2Place 等精确几何类任务下降。

因此最稳妥的总结是：

> 该流程提高了结构化空间推理和规划样本的信噪比，但没有证明数据在所有能力上都更优。后续需要为坐标、掩膜和仿真可执行性增加任务专用校验器，并用人工校准集评估 Judge 的一致性。

如果当时没有完成系统性的人工抽检，不要在面试中编造抽检比例。可以坦诚说明主要依赖规则和模型过滤，并将人工校准集作为后续改进。

## 2. 这些大型数据集不是已经有人专门过滤了吗？为什么还要再过滤？

### 面试主回答（约 30 秒）

是的，大型公开数据集发布前通常已经有标注规范、人工复核和自动校验，因此我们原则上信任其官方原始标注。我们做的不是重复判断原始标注是否正确，而是对经过模板转换、结构化 CoT 扩展和跨数据集映射后生成的训练样本进行二次质检。

原始数据的质量保证只覆盖它原本定义的任务。例如 LVIS 保证的是物体类别和实例 mask，不能保证从 mask 推导出的抓取点或规划 CoT 正确；EPIC-KITCHENS 保证的是动作片段及 verb/noun 标注，不能保证我们扩展出的多步计划正确；BridgeData 保证轨迹在原机器人环境中成立，不能保证映射到 AI2-THOR 后仍满足对象、动作和物理约束。

所以我们的工作更准确地说是 **conversion validation + task-specific filtering**，不是重新清洗整个原始数据集。

### 为什么源数据经过质检后仍需要二次验证？

1. **派生错误**：模板或 VLM 可能在原标注之外补充不存在的物体、关系或动作。
2. **任务错配**：原标注对检测/分割任务正确，但不一定能支持空间推理、可供性或任务规划。
3. **跨域差异**：真实机器人轨迹映射到仿真环境后，对象集合、坐标系、动作接口和物理条件可能不同。
4. **格式与训练约束**：数据可能语义正确，但不满足 JSON schema、坐标归一化、多选题输出等训练要求。
5. **分布问题**：单条样本都正确，整体仍可能高度重复、类别失衡、模板泄漏或与评测集近重复。

### 可直接背诵的完整回答

> 大型数据集本身确实有专业的质检流程，所以我们信任官方原始标注。我们评分的主要对象不是原始数据，而是经过任务转换后的 `instruction-input-output` 样本。因为我们将关系三元组、实例 mask、视频动作片段和机器人轨迹进一步转换为空间问答、结构化 CoT 和多步计划，这些派生内容已经超出了原数据集的质量保证范围。我们重点检查派生结论是否仍有视觉或源标注支撑、计划是否可执行、格式是否符合训练要求，并处理重复、能力分布和跨环境映射问题。因此这是下游任务适配和二次质检，而不是认为大型数据集本身错误很多。

### 不要这样回答

- “大型数据集错误很多，所以我们重新给它们纠错。”
- “语义相似度高就说明答案正确。”
- “Skywork-VL Reward 的分数就是 1–6 分。”
- “Qwen3-VL 比原数据集标注员更可靠，所以以模型判断为准。”
- “筛选后所有评测指标都提升了。”
