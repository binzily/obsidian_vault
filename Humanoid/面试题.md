# Humanoid RL 面试题

## 1. 如何理解“将策略从显式 gait clock 驱动，转向基于命令条件、接触事件与足端几何约束的 gait-free 学习框架”？

**回答：**

这里的核心变化是：我不再把固定步态相位直接作为策略的主要**驱动**信号，而是让策略根据当前任务命令、真实接触事件和足端运动约束自己学出合理步态。

“命令条件”指策略和奖励都围绕当前 command 变化。本项目的 move-operate command 是 6 维：

```text
[vx, vy, yaw_rate, roll, pitch, height]
```

因此策略不仅要跟踪前进、侧移、转向速度，还要同时满足躯干 roll、pitch 和高度目标。速度命令大时，奖励鼓励稳定行走；速度为 0 时，奖励鼓励站稳；roll/pitch/height 非中性时，奖励鼓励身体姿态跟踪。大姿态操作时，还会通过 pose-speed gate 降低可用速度上限，避免在大幅俯仰或下蹲时仍然要求高速行走。

“接触事件”指用**脚实际接触地面的事件来塑造步态**，而不是用固定相位表硬性规定左右脚何时摆动。具体包括：触地、离地、空中时间、支撑时间、左右脚是否交替、是否双脚同时触地、是否飞起等。比如 gait-free 中的步态周期奖励会看 touchdown 事件之间的时间差，从完整步态事件中判断节奏是否合理。

“足端几何约束”指直接约束脚的位置、朝向和运动形态是否合理。典型约束包括：步宽不能太窄或太宽，左右脚不能交叉，侧移时双脚前后错位不能过大，脚尖 yaw 不能明显内八或外八，摆动脚高度要在合理区间，落地速度不能太大，支撑脚不能滑。

所以 gait-free 不是完全不管步态，而是不把显式 `sin/cos(gait_phase)` 喂给 policy，也不靠固定 phase mask 强制左右脚交替。它通过接触事件、空中时间、落地事件和足端几何约束，让 policy 自己学出满足命令的自然步态。

## 2. 如何理解 sim2sim 仿真以及部署验证链路中的传感器、状态估计、观测构造、策略推理、PD 下发闭环？

**回答：**

这个闭环可以概括为：

```text
MuJoCo 或真机传感器
-> RobotData 状态缓存
-> 状态估计与观测构造
-> ONNX policy 推理
-> 目标关节位置 + kp/kd
-> CmdMotorCtrl 下发
-> 底层 PD 产生力矩
-> 机器人或仿真动力学响应
-> 下一帧传感器反馈
```

在 sim2sim 中，MuJoCo 模拟真机底层接口：它发布 `/leg/status`、`/waist/status`、`/arm/status` 和 `/imu/status`，同时订阅 `/leg/cmd_ctrl`、`/waist/cmd_ctrl`、`/arm/cmd_ctrl`。这样 `xmigcs` 控制节点看到的接口和真机部署基本一致。

传感器侧主要包括两类数据。第一类是电机状态，包括关节位置、关节速度、电流或力矩估计、温度等；第二类是 IMU 状态，包括 yaw/pitch/roll、角速度和线加速度。控制程序把这些数据写入 `RobotData`，形成当前控制周期的机器人状态缓存。

状态估计阶段会基于 IMU 姿态计算 `projected_gravity`，并读取角速度、关节位置、关节速度等可部署观测。对于 move-operate，actor 观测主要包括：

```text
ang_vel
projected_gravity
command
joint_pos - default_joint_pos
joint_vel - default_joint_vel
last_action
```

如果是 gait-clock 版本，还会额外拼接：

```text
sin(gait_phase)
cos(gait_phase)
phase_ratio
```

gait-free 版本不把显式相位拼进 actor 观测。

策略推理阶段使用导出的 ONNX policy。policy 输出的是 action，经过缩放和平移后变成目标关节角：

```text
target_joint_pos = action * action_scale + default_joint_pos
```

然后控制节点把目标关节角写入 `q_d`，目标速度通常写 0，前馈力矩通常写 0，同时写入每个关节的 `kp/kd`。这些命令通过 `CmdMotorCtrl` 发给 MuJoCo 或真机底层控制器。

最后，底层 PD 根据目标位置、实际位置、目标速度、实际速度和前馈力矩计算实际输出力矩，驱动机器人运动。下一帧传感器再反馈新的关节状态和 IMU 状态，闭环继续运行。

## 3. 发给机器人控制器的是转矩还是目标位置？

**回答：**

对当前 move-operate 策略来说，本质上发的是目标关节位置，而不是 policy 直接输出转矩。

消息字段里确实同时包含：

```text
pos, spd, tor, kp, kd
```

但当前 move-operate 的策略输出 action 后，会转换成目标关节角 `q_d`；目标速度 `q_dot_d` 写 0；前馈力矩 `tau_d` 写 0。因此策略本身不是 torque policy，而是 position target policy。

底层实际执行时，会用 PD 公式产生力矩：

```text
torque = kp * (pos_d - pos_a) + kd * (spd_d - spd_a) + tor_d
```

其中 `pos_d` 是策略给出的目标关节角，`pos_a` 是实际关节角，`spd_d` 通常为 0，`spd_a` 是实际关节速度，`tor_d` 当前通常为 0。

所以更准确的说法是：策略输出目标关节位置；控制节点下发位置、速度、前馈力矩和 PD 增益；最终由底层控制器或 MuJoCo 根据 PD 计算实际力矩。

## 4. AMP 的原理是什么？

**回答：**

AMP 是 Adversarial Motion Priors，本质上是在强化学习任务奖励之外，引入一个对抗式运动先验，让策略运动更像专家数据。

它不是让机器人逐帧跟踪专家轨迹，也不是简单做关节角 L2 imitation loss。AMP 会训练一个 discriminator，用来区分两类 transition：

```text
专家运动 transition:  (s_t, s_{t+1})
策略运动 transition:  (s_t, s_{t+1})
```

discriminator 的目标是判断一段状态转移更像专家还是更像当前策略。训练时，专家数据被打上正标签，策略 rollout 数据被打上负标签。与此同时，policy 会通过 PPO 优化，让自己产生的 transition 更容易被 discriminator 判断为专家运动。

在本项目的 move-operate 中，AMP observation 主要使用腿部关节位置、腿部关节速度，以及左右脚相对 root 的位置。因此 AMP 主要提供“走起来像专家 walk 数据”的运动风格先验。

任务本身的速度跟踪、roll/pitch/height 姿态控制、站稳、脚不交叉、落地不要太重、支撑脚不要滑等，仍然主要由 task reward 和 command 条件负责。AMP 更像是运动自然性和步态风格的约束，而不是替代任务奖励。

## 5. 随机种子不同与梯度不同步分别会导致什么问题？

**回答：**

这两个问题性质不同，不能混在一起看。

随机种子不同首先要区分是环境 seed 不同，还是模型初始化 seed 不同。环境 seed 不同在多卡训练中通常是合理的，因为每个 rank 采到不同 rollout，数据会更多样。比如我在分布式训练中会让不同 rank 使用不同环境 seed，但保持模型 seed 一致。

模型初始化 seed 不同则更危险。如果初始化之后会从 rank0 broadcast policy 和 discriminator 参数，那么问题不大，因为所有 rank 最终会从同一套参数开始训练。但如果**没有 broadcast**，不同 rank 从一开始就是不同模型，即使后面做梯度 all-reduce，平均的也是不同参数点上算出来的梯度，数学意义已经不对。

梯度不同步比随机种子问题更严重。正确的多卡训练流程应该是：

```text
每张卡各自采样
每张卡各自计算 loss 和 grad
所有 rank 对 grad 做 all_reduce 求平均
每张卡用同一个平均 grad 更新同一份参数
```

如果梯度完全没有同步，每张卡的 policy 会越训越不一样，rank0 保存的模型只代表 rank0，其他卡采样用的是另外的策略，训练曲线会非常乱。

如果梯度同步了但没有除以 **`world_size`**，等效学习率会被放大 `world_size` 倍。常见现象是单卡正常，多卡后 KL 变大、动作发散、value loss 异常、action std 异常，最后表现为不收敛。

所以总结一下：

```text
环境 seed 不同：通常是正常的，用来增加采样多样性
模型 seed 不同但有参数 broadcast：通常可以接受
模型 seed 不同且没有 broadcast：起点不同，训练不严格正确
梯度没同步：各卡模型分叉，训练基本错误
梯度同步但没平均：等效学习率变大，容易发散
```

## 6. 如果单卡正常但是多卡不收敛，应该怎么定位问题？

**回答：**

我会先排除分布式训练链路问题，而不是直接改 reward 或调 curriculum。因为单卡正常、多卡不收敛，优先怀疑的是同步、归一化、采样分布和 batch 尺度变化。

第一步，检查**参数**是否真的同步。每次 update 后，在每个 rank 打印 policy 和 discriminator 参数的 checksum。如果 checksum 不一致，说明 broadcast 或 gradient all-reduce 有问题。对于 AMP 训练，policy 和 discriminator 都要查，因为 AMP reward 来自 discriminator。

第二步，检查 **normalizer** 是否同步。多卡 PPO/AMP 中，obs normalizer、privileged obs normalizer、amp normalizer 都可能影响输入分布。如果每个 rank 的 normalizer 统计不同，那么同一个物理状态会被归一化成不同输入，梯度平均就会变脏。现象是参数 checksum 可能一致，但训练仍然不稳。

第三步，检查每个 rank 的 rollout 分布。分别打印每个 rank 的 reward 均值、episode length、command 分布、termination rate、obs mean/std。如果某张卡明显异常，问题更可能在环境采样、domain randomization、seed、仿真状态或某个 GPU 上，而不是梯度同步。

第四步，检查 batch size 和学习率尺度。多卡后总采样量变成：

```text
num_steps_per_env * num_envs * world_size
```

即使梯度平均是正确的，PPO 的 minibatch size、num_mini_batches、num_learning_epochs、KL adaptive learning rate 也可能改变实际更新强度。如果 KL、entropy、value loss 在多卡明显变坏，就要重点检查这些训练超参。

第五步，专门检查 AMP 指标。对于 AMPPPO，需要看：

```text
amp_policy_pred
amp_expert_pred
amp_loss
grad_pen_loss
AMP reward
task reward
```

如果 discriminator 很快把 expert 和 policy 分得过开，或者 AMP reward 尺度压过 task reward，多卡训练会比单卡更容易被 AMP 带偏。

我通常用下面这个判断顺序：

```text
参数不同 -> 梯度同步或参数广播问题
参数相同但 normalizer 不同 -> 归一化统计问题
参数和 normalizer 都正常但某个 rank rollout 异常 -> 环境、seed 或 domain randomization 问题
这些都正常但 KL/entropy/value loss 异常 -> batch、学习率或 PPO 超参问题
AMP 指标异常 -> discriminator、amp_normalizer 或专家采样问题
```

## 7. 为什么部署代码里已经下发 `tor`，底层还要再按 PD 公式计算 torque？

**回答：**

关键点是：部署代码下发的 `tor` 不是“最终电机力矩”，而是前馈力矩 `tor_d`。底层控制接口收到的是一组关节控制命令：

```text
pos_d, spd_d, tor_d, kp, kd
```

底层控制器再结合实际关节状态计算最终输出力矩：

```text
output_torque = kp * (pos_d - pos_a) + kd * (spd_d - spd_a) + tor_d
```

所以最终力矩由三部分组成：位置误差项、速度阻尼项、前馈力矩项。`tor_d` 是加在 PD 输出上的补偿项，不是用来替代 PD 的。

在我当前的 move-operate 里，policy 只输出 action，也就是目标关节位置偏移。FSM 会把它转换成目标关节角：

```text
pos_d = action * action_scale + default_joint_pos
```

而 `spd_d` 和 `tor_d` 不是 policy 输出的。它们由 FSM 固定写入：

```text
spd_d = 0
tor_d = 0
```

这里的 `spd_d` 是期望关节速度，不是实际关节速度。它进入 PD 公式里的 D 项：

```text
kd * (spd_d - spd_a)
```

在当前 move-operate 里，policy 没有输出期望速度，FSM 每个控制周期都把 `spd_d` 写成 0。因此 D 项实际变成：

```text
kd * (0 - 当前关节速度)
```

它的作用就是阻尼，抑制关节速度和高频抖动，而不是让关节跟踪某条显式的速度轨迹。

`kp/kd` 来自部署配置文件。也就是说，当前 move-operate 实际上是“policy 输出目标关节位置，底层 PD 跟踪目标位置”的位置控制模式，最终力矩近似为：

```text
output_torque = kp * (目标关节角 - 当前关节角) + kd * (0 - 当前关节速度)
```

这里 `tor_d` 是每个控制周期都为 0，不是只有第一次为 0。但这不代表最终电机力矩为 0，因为只要目标关节角和当前关节角不同，或者当前关节速度不为 0，PD 项就会产生力矩。

`xsim_mujoco` 里还要再算一次 torque，是因为 MuJoCo 的 motor actuator 接收的是实际力矩。sim2sim 要模拟真机底层控制器，所以需要先把上层下发的 `pos_d/spd_d/tor_d/kp/kd` 转成最终 torque，再写入 MuJoCo。

`tor_d` 不为 0 的情况通常是上层控制器想加前馈力矩，比如重力补偿、动力学前馈、接触力控制，或者策略本身输出力矩：

```text
tor_d = gravity_compensation(q)
tor_d = M(q) qdd_des + C(q, qd) qd + G(q)
tor_d = J(q)^T F_des
tor_d = policy_action
```

如果我要做真正的直接力矩控制，通常会让：

```text
kp = 0
kd = 0
tor_d = policy 或控制器输出的力矩
```

也可以使用混合控制：

```text
pos_d = policy_pos
tor_d = policy_torque
kp/kd > 0
```

所以我在面试里会强调：当前 move-operate 不是直接力矩控制，而是位置目标 + PD 控制；`spd_d` 和 `tor_d` 当前都由 FSM 固定置零，最终力矩由底层根据 `pos_d/spd_d/tor_d/kp/kd` 和实际关节状态实时计算。

## 8. move-operate 里的 command 课程学习具体做了什么？哪些奖励在前期被课程学习压低？

**回答：**

我这里的 move-operate 里其实有两层相关的课程学习：一层是命令本身的课程学习，另一层是 pose 奖励权重的课程学习。它们都围绕 `roll / pitch / height` 这组三维 operate 姿态命令，而不是围绕速度命令本身。

第一层是 R/P/H command curriculum。训练时完整 command 是：

```text
[lin_vel_x, lin_vel_y, ang_vel_z, roll, pitch, height]
```

前期并不是一上来就给完整幅度的 `roll / pitch / height`，而是用 progress 从中性姿态逐渐放开：

```text
roll   = raw_roll * progress
pitch  = raw_pitch * progress
height = neutral_height + progress * (raw_height - neutral_height)
```

所以前期更像是在先学稳定行走，`roll / pitch / height` 的操作难度被压低；后期 progress 到 1 之后，才变成完整的 6D command 跟踪。

第二层是 pose reward scale curriculum。环境里会根据同一个 progress 生成：

```text
pose_reward_scale = pose_reward_floor + (1 - pose_reward_floor) * progress
```

当前配置里 `pose_reward_floor = 0.25`，所以 warmup 阶段 pose 类奖励只有 25% 权重，之后线性涨到 100%。

被这个 `pose_reward_scale` 直接乘到的奖励主要是：

```text
track_roll_pitch_exp
track_roll_pitch_body_exp
track_pelvis_roll_pitch_fraction_exp
track_base_height_exp
base_height_error_l1
```

在当前 dex move-operate 配置中，实际启用的对应项是：

```text
track_roll_pitch_exp              # 实际跟踪 waist_pitch_link 的 roll/pitch
track_pelvis_roll_pitch_exp       # pelvis 跟踪一部分 roll/pitch
track_height_exp                  # base height 跟踪
base_height_error_l1              # base height 误差惩罚
```

没有被这个 pose reward scale 直接压低的包括速度跟踪、gait-clock 步态相位奖励、gait-free 的接触事件/足端几何约束、energy、action rate、dof acc、termination、alive 和 AMP reward。也就是说，前期不是把所有奖励都变小，而是专门把 operate 姿态相关的目标和奖励放缓。

按当前配置换算：

```text
warmup_steps = 192000 env steps
ramp_steps   = 480000 env steps
num_steps_per_env = 24
```

所以大约是：

```text
前 8000 iter：progress = 0，R/P/H 命令基本中性，pose_reward_scale = 0.25
8000 到 28000 iter：R/P/H 命令和 pose_reward_scale 线性放开
28000 iter 之后：progress = 1，完整 6D command 和完整 pose reward
```

总结起来，我会说：这个课程学习的核心思想是 walk-first。先让策略把速度跟踪、步态和站稳学起来，再逐渐增加 `roll / pitch / height` 操作姿态的命令难度，并同步增强对应的 pose tracking 奖励。

## 9. symmetry 在双足强化学习里的原理是什么？

**回答：**

symmetry 的核心思想是利用机器人左右结构对称性。对于双足机器人来说，一个“左脚支撑、右脚摆动”的状态，左右镜像后应该对应“右脚支撑、左脚摆动”的状态；这两个状态在物理上是等价的，策略输出也应该按同样规则镜像。

因此 symmetry 通常做两件事：data augmentation 和 mirror loss。

data augmentation 是把原始训练样本：

```text
obs, action
```

扩展成：

```text
obs, action
mirror(obs), mirror(action)
```

这样一条左侧经验可以自动生成一条右侧经验，提高样本效率，也减少策略只偏向某一侧腿的风险。

mirror loss 则是直接约束策略满足：

```text
policy(mirror(obs)) ≈ mirror(policy(obs))
```

也就是说，如果输入状态被左右镜像，策略输出也应该是原输出的左右镜像版本。这个约束会让 policy 学到更对称、更一致的左右腿控制规律。

在我当前的 move-operate 训练里，镜像主要包括：

```text
左腿和右腿关节交换
左右相关的符号取反，例如 lateral velocity、yaw、roll、hip roll、hip yaw、ankle roll 等
gait-clock 版本里左右脚相位交换
critic observation 里的左右脚接触状态交换
action 也按同样规则做左右交换和符号取反
```

例如 action 里左腿 6 个关节和右腿 6 个关节会互换；左右镜像后方向相反的关节会取负号；腰部里 `waist_roll` 会取负号，而 `waist_pitch` 不取负号。

所以我会这样总结：symmetry 不是额外的任务奖励，而是把“左侧动作经验”和“右侧动作经验”绑定起来，通过镜像数据增强和 mirror loss 让策略满足左右一致性。它的作用是提高样本效率、减少左右偏置，并让步态和操作动作更对称。

## 10. xmigcs 中能看到哪些传感器？move-operate 实际用到了哪些？

**回答：**

我会先区分两件事：`xmigcs` 能接入的原始状态/输入源，和当前 `MOVEOPERATE` policy 实际放进 actor observation 的量。它们不是完全一样的。

`xmigcs` 底层能看到的核心机器人状态主要有两类。第一类是电机状态，包括腿、腰、手臂、头部一共 31 个电机的 `pos`、`speed`、`current`、`temperature` 和 `error`。这些会被写进 `RobotData` 里的 `q_a_`、`q_dot_a_`、`tau_a_`、`temperature_a` 和 `diagnostic_info`。这里要注意，`tau_a_` 不是直接力矩传感器读数，而是由电流乘以 `ct_scale` 换算出来的力矩估计。

第二类是 IMU 状态，包括：

```text
yaw / pitch / roll
gyro_x / gyro_y / gyro_z
acc_x / acc_y / acc_z
```

其中 actor 常用的 `projected_gravity` 不是 IMU 直接给的字段，而是根据 IMU 姿态把世界系重力投影到机体系里算出来的。

除了机器人本体传感器，`xmigcs` 还会接收控制输入，例如 SBUS 手柄 `/sbus_data`、Xbox `/xbox_data`，以及上层 topic 里的速度命令、高度命令、姿态命令、足点命令和 FSM 切换命令。这些是控制输入，不是机器人本体传感器，但会参与构造策略命令。

对我当前的 `MOVEOPERATE` 来说，actor observation 实际使用的是：

```text
IMU 角速度
IMU 姿态推出来的 projected_gravity
command: lin_vel_x, lin_vel_y, ang_vel_z, roll, pitch, height
14 个 action 关节的位置和速度
上一帧 action
gait-clock 版本额外使用 gait phase 的 sin/cos 和 phase_ratio
```

这里的 14 个 action 关节是两条腿 12 个关节，加上 `waist_roll` 和 `waist_pitch`。`waist_yaw` 当前不是 policy 输出控制的自由度，而是在 FSM 里保持到固定位置；手臂也是 hold 住，不进入 move-operate actor 的 action。

当前代码里没有看到 move-operate 直接使用足底力传感器、真实接触传感器、相机、雷达、电机温度、电机 error、头部状态或手臂状态作为 actor 输入。接触相关逻辑更多是动力学模型基于关节状态、估计力矩和足端高度推出来的，而且 FSM 里的接触保护目前还是注释状态。

所以面试里我会这样总结：部署侧能看到的是电机状态、IMU 和外部命令输入；但 move-operate 策略真正吃进去的是可部署的 proprioception，也就是 IMU 角速度、重力投影、关节位置/速度、命令和上一帧动作。它不是依赖仿真里才能直接读到的 base velocity、接触力或足底 contact flag。

## 11. 训练里哪些东西真机不能直接读？真机是怎么处理的？

**回答：**

我会先把训练里的量分成两类：一类是部署可获得的 actor observation，另一类是仿真里才能直接拿到的 privileged information。真正部署时，policy 只能依赖第一类；第二类主要用于训练 reward、critic 或日志，不能直接喂给真机 actor。

当前 move-operate 的 actor observation 基本是部署可获得的：

```text
IMU 角速度
IMU 姿态推出来的 projected_gravity
命令: lin_vel_x, lin_vel_y, ang_vel_z, roll, pitch, height
关节位置和关节速度
上一帧 action
gait-clock 版本额外有软件维护的 phase
```

这些在真机上分别来自 IMU、电机编码器、手柄或上层 topic，以及控制器自己的历史缓存。`projected_gravity` 不是传感器直接给的字段，而是用 IMU 姿态把重力方向投影到机体系得到的。

真机不能像仿真一样直接读的主要是这些：

```text
base/root linear velocity
base/root 在世界系的精确位姿
足底 contact flag
足底接触力 / net contact force
足端世界系位置、速度、高度、yaw
各 link 的 body_state_w / body_lin_vel_w
critic 里的 feet_contact、root_lin_vel、arm_target_offset 等特权信息
奖励函数里用到的精确碰撞、足端几何、落脚瞬间、air time 等事件量
```

以 `base linear velocity` 为例，Isaac Lab 里可以直接读 `root_lin_vel_b`，所以训练时 critic 或速度奖励可以直接用。但真机没有一个传感器能直接给出完全可靠的机体系 base 线速度。部署时的处理是：不要把它放进 actor observation。策略只根据 IMU、关节状态和命令闭环控制；如果工程上需要速度状态，可以再用 IMU、运动学、足端接触假设或外部定位做估计，但这不是当前 move-operate actor 的必要输入。

以足底接触为例，仿真里 `contact_sensor` 可以直接给足底接触力和接触标志，gait-free 训练里的很多奖励都依赖它，比如接触交替、air time、touchdown、no slip、足端高度和接触力变化。但真机这里没有直接的足底力/接触传感器输入。部署时同样不是把 contact flag 喂给 actor，而是在训练中用这些奖励塑造策略，部署时让策略只凭可获得的本体感知执行。如果安全保护或状态估计需要接触，可以用电机电流/力矩估计、关节运动学和动力学模型去估计接触，但这属于部署侧估计逻辑，不是 actor 的直接观测。

以足端位置、高度、yaw 为例，仿真可以直接读 link 的世界系状态，因此奖励里可以精确约束步宽、足 yaw、足端高度、落脚速度等。真机不能直接读这些 link state。处理方式是训练时把它们作为 reward shaping，让策略学出合理足端行为；部署 actor 不依赖这些量。必要时可以通过编码器和 URDF 正运动学估计足端相对机身的位置，但不会像仿真一样直接得到干净的世界系真值。

所以我会这样总结：训练里确实用了很多仿真特权信息，但它们主要用于 critic 和 reward，而不是部署 actor。真机部署的关键原则是 actor observation 必须只包含可部署量：IMU、关节编码器、命令和上一帧动作。仿真特权量的作用是在训练时塑形，不是在真机上直接读取。

## 12. 环境镜像是怎么写的？如果 observation 里有加速度，应该怎么镜像？

**回答：**

我这里的 symmetry 主要用于 data augmentation 和 mirror loss。做法是把原始样本：

```text
obs, action
```

扩展成：

```text
obs, action
mirror(obs), mirror(action)
```

move-operate 的镜像规则是按 observation layout 手工写的。gait-free 单帧 actor observation 是：

```text
ang_vel: 3
projected_gravity: 3
command: 6
joint_pos: 14
joint_vel: 14
last_action: 14
```

所以镜像时分三类处理。

第一类是空间向量和命令。左右镜像等价于过机器人矢状面对称，所以左右方向相关的量要变号。例如 `projected_gravity_y`、`lin_vel_y`、`yaw_rate`、`roll` 会取负；角速度作为旋转量，`ang_vel_x` 和 `ang_vel_z` 取负，`ang_vel_y` 不变。

第二类是关节量。双腿关节要左右交换，同时方向相反的关节要取负。比如 `hip_roll`、`hip_yaw`、`ankle_roll`、`waist_roll` 取负；`hip_pitch`、`knee`、`ankle_pitch`、`waist_pitch` 不取负。这个规则会同时用于 `joint_pos`、`joint_vel`、`last_action` 和 policy 输出 action。

第三类是 critic 或 gait-clock 里的额外量。critic 里的 `feet_contact` 左右交换，`root_lin_vel_y` 取负，手臂 nuisance 按左右手臂交换和关节方向取负。gait-clock 版本里的 `sin phase`、`cos phase`、`phase_ratio` 只做左右脚交换，不做符号取反。

当前 move-operate 的 observation 里没有加速度，所以代码里没有单独写 acceleration mirror。`joint_acc_l2`、`body_ang_acc_xy_l2` 这些加速度项主要是 reward penalty，不参与 actor/critic 的 mirror data augmentation。

如果以后把加速度放进 observation，我会按物理量类型处理：线加速度是普通线性向量，左右镜像时 `y` 分量取负；角加速度和角速度一样是旋转量，`roll/yaw` 对应的分量取负，`pitch` 分量不变；关节加速度则和关节位置、速度、action 一样，左右交换并按关节轴方向决定是否取负。

所以我会记住这句话：**线性量按“左右方向 y 取负”；旋转量按“roll/yaw 取负，pitch 不变”；关节量按“左右交换 + 方向相反的关节取负”**。

## 13. AMP 判别器用了哪些额外状态信息？需要加噪声吗？

**回答：**

我会先强调，AMP 判别器用的不是 actor policy 的完整 observation，而是一套专门为模仿专家运动构造的 AMP observation。它只在训练时参与判别和产生 AMP reward，部署时不会进入真机控制链路。

在当前 move-operate 里，判别器看到的是 30 维腿部运动状态：

```text
右腿关节位置 6
左腿关节位置 6
右腿关节速度 6
左腿关节速度 6
左脚相对 root 的位置 xyz 3
右脚相对 root 的位置 xyz 3
```

这里的足端位置不是世界系绝对位置，而是先用足端世界位置减去 root 位置，再通过 root 的逆四元数转到机器人 root/body 坐标系下。因此 AMP 判别器主要判断的是：当前策略生成的腿部关节运动和双脚相对身体的几何运动，像不像专家 walk 数据。

专家 motion 文件里原始是 52 维：

```text
20 维 joint_pos + 20 维 joint_vel + 12 维 end_effector_pos
```

但训练配置通过 `amp_observation_indices` 只从专家数据里抽出和当前 env AMP observation 对齐的 30 维。所以 AMP 不是拿全部专家状态做判别，而是只看和当前任务相关的腿部关节和足端运动子空间。

关于噪声，我的结论是：默认不需要，也不建议直接给 AMP observation 加传感器噪声。

原因有三点。第一，AMP 判别器是训练时的风格奖励模块，不是部署时的 actor 输入，真机上不会用判别器。第二，AMP 可以使用仿真里的特权运动学状态，比如足端相对 root 的位置，因为它只用于训练约束，不要求真机能直接读取。第三，如果只给 policy 侧 AMP observation 加噪声，而专家 motion 不加噪声，判别器可能学到的是“有没有噪声”的差异，而不是动作风格差异。

所以我会这样区分：actor observation 可以加 IMU、关节位置、关节速度等传感器噪声，用来提升部署鲁棒性；AMP observation 通常保持干净，并通过 normalizer 和 gradient penalty 稳定判别器训练。如果以后真的要做 AMP 数据增强，也应该对 policy AMP 样本和 expert AMP 样本一致地加小噪声，而不是只加一边。

## 14. ROS 如何自定义消息？创建 ROS 功能包的流程是什么？这套链路里定义了哪些节点？

**回答：**

我会先把 ROS 功能包分成两类：一类是只定义接口的消息包，另一类是实际运行节点的业务包。自定义消息一般应该单独放在消息包里，比如 `bodyctrl_msgs`；节点代码再依赖这个消息包。

**自定义消息的流程**是：先创建一个 `ament_cmake` 包，例如 `bodyctrl_msgs`，然后在包里建立 `msg/` 目录，写 `.msg` 文件，例如：

```text
SbusData.msg
CmdMotorCtrl.msg
MotorStatusMsg.msg
Imu.msg
```

接着在 `package.xml` 里加入 `rosidl_default_generators`、`rosidl_default_runtime`，并声明它属于 `rosidl_interface_packages`。在 `CMakeLists.txt` 里用 `rosidl_generate_interfaces()` 把这些 `.msg` 注册进去。编译后，C++ 里就可以 `#include <bodyctrl_msgs/msg/sbus_data.hpp>`，Python 里就可以 `from bodyctrl_msgs.msg import SbusData`。

**创建普通 ROS2 功能包**的流程是：先用 `ros2 pkg create` 创建包，然后在 `src/` 下写节点，在 `CMakeLists.txt` 里 `find_package()` 依赖，用 `add_executable()` 注册可执行文件，用 `ament_target_dependencies()` 链接依赖，再通过 `install(TARGETS ...)` 安装节点。如果需要统一启动，就再写 `launch/*.launch.py`。最后用 `colcon build` 编译，并 `source install/setup.bash` 让环境能找到新包和新消息。

以我当前这套 sim2sim 链路为例，主要有三个节点。

第一个是 `usb_sbus` 节点。它在 `usb_sbus` 功能包里，C++ 可执行文件是 `usb_sbus_node`，launch 后节点名是 `usb_sbus`。它负责从 SBUS/串口读手柄数据，然后发布：

```text
/sbus_data        sensor_msgs/Joy
/sbus_data/event  bodyctrl_msgs/SbusData
```

第二个是 `xmigcs_control_node`。它是 `xmigcs` 里的 Python ROS2 控制节点，仿真时跑的是 `rl_control_node_sim.py`，本质上继承真实控制节点，只是把配置改成 sim。它负责 FSM 状态机、policy 推理、手柄或 topic 命令解析，并向底层发布电机控制命令。它常用的订阅包括：

```text
/sbus_data
/hric/robot/cmd_vel
/hric/robot/walk_height_cmd
/hric/robot/fsm_state_cmd
/leg/status
/waist/status
/arm/status
/imu/status
```

它常用的发布包括：

```text
/leg/cmd_ctrl
/waist/cmd_ctrl
/arm/cmd_ctrl
/hric/robot/rl_state
```

第三个是 MuJoCo 仿真节点 `mujoco_simulator_dex`。它由 `xsim_mujoco/scripts/simulator_view_asyn.py` 创建，作用是模拟底层机器人控制器。它订阅 `xmigcs_control_node` 发出的电机命令，在 MuJoCo 里执行 PD/动力学仿真，然后发布模拟出来的电机状态和 IMU。默认 `--comm ros2` 模式下，它订阅：

```text
arm/cmd_ctrl
leg/cmd_ctrl
waist/cmd_ctrl
```

并发布：

```text
arm/status
leg/status
waist/status
imu/status
robot/abs_position
waist_pitch_link/abs_position
```

所以我会这样总结：自定义消息解决“通信数据结构”的问题，功能包解决“节点如何编译、安装、启动”的问题；在我这套部署/仿真链路里，`usb_sbus` 负责手柄输入，`xmigcs_control_node` 负责状态机和策略控制，`mujoco_simulator_dex` 负责仿真底层机器人并回传状态。

## 15. 机器人的运动学、动力学和雅各比矩阵分别是什么？

**回答：**

我会先把这三个概念分清楚：运动学只关心几何关系，动力学关心力和运动的关系，雅各比矩阵是连接关节空间和任务空间的重要桥梁。

运动学研究的是：给定机器人关节角，身体、腰部、足端、手端这些 link 在哪里、姿态是什么、速度是多少。它不关心质量、惯量和力矩。比如给定髋、膝、踝关节角，算脚底相对机身的位置，就是正运动学；反过来，给定脚底目标位置，求对应的关节角，就是逆运动学。

可以简单写成：

```text
q -> foot pose / link pose
qdot -> foot velocity / link velocity
```

雅各比矩阵是运动学里很核心的量。它描述的是：关节速度如何映射成末端速度。

```text
v_foot = J(q) qdot
```

这里 `q` 是关节位置，`qdot` 是关节速度，`J(q)` 是当前姿态下的雅各比矩阵，`v_foot` 是足端线速度和角速度。直观理解就是：每个关节动一点，会让脚往哪个方向动多少。

雅各比还有一个非常重要的力学转置关系：

```text
tau = J(q)^T F
```

这表示足端受到的力 `F`，可以通过 `J^T` 映射成关节力矩 `tau`。所以雅各比不仅能把关节速度映射到足端速度，也能把足端力映射到关节力矩。对人形机器人走路来说，脚和地面的接触力最后就是通过接触雅各比影响各个腿部关节。

动力学研究的是：电机力矩、接触力、重力、质量和惯量如何决定机器人的加速度和运动。常见形式是：

```text
M(q) qddot + h(q, qdot) = tau + J_c(q)^T lambda
```

其中 `M(q)` 是质量/惯量矩阵，`qddot` 是关节加速度，`h(q, qdot)` 包含重力、科氏力和离心力，`tau` 是电机力矩，`J_c(q)^T lambda` 是接触力通过接触雅各比映射到关节空间后的广义力。

放到我的 half-walk / move-operate 里，可以这样理解：policy 输出目标关节位置，底层 PD 根据目标位置、当前关节位置和速度算出实际力矩；这些力矩和地面接触力一起进入机器人动力学，决定下一时刻身体和关节怎么运动。而运动学和雅各比可以用来计算足端位置、足端速度、接触力对关节力矩的影响，或者做动力学模型里的接触估计。

所以面试里我会这样总结：运动学回答“关节角对应机器人在哪里、怎么动”，动力学回答“力矩和接触力会让机器人怎么加速”，雅各比矩阵回答“关节空间和任务空间之间如何互相映射”。它一方面有 `v = J qdot` 的速度映射，另一方面有 `tau = J^T F` 的力映射。

## 16.base velocity 在真机中怎么得到的？
