> 可以把 PPO 训练想成这个流程：
> 
> 1. 跑环境，采一大批 rollout 数据
> 2. 用这批数据更新策略
> 3. 进入下一轮
> 
> 这里对应关系就是：
>

## 1.TienKung-Lab/legged_lab

**如果要设计一个RSL-RL，怎么设计？**
1. 从环境拿 observation
2. 用 policy 网络输出 action
3. 把 action 喂回环境
4. 收集一段 rollout
5. 用 PPO/AMP-PPO 更新网络参数
6. 重复上面过程
先有RSL-RL Runner在有BaseEnv环境，这样顺着看，也就知道env需要怎么写了：给obs，做action。

**如果要设计一个BaseEnv环境，怎么设计？**
- 仿真层
- 实体层
- 状态层
- 指令层
- 控制循环层

**如果要设计一个deploy pipeline，怎么设计？**
1. GetState(...) 读当前机器人状态
2. RunFSM(...) 决定现在该停、归零，还是跑策略
3. SetCommand(...) 把目标命令下发到底层
robot_interface可以拿到机器人传感器的状态给policy，也可以拿到policy的action下发命令给机器人。
具体细节如下：
**先建立一个最小心智模型**

部署本质上不是“一个大 C++ 项目”，而是“一个 400Hz 实时控制循环，加几个功能模块”。
最核心的入口在 main.cpp。每个周期只干 3 步：

1. `GetState(...)` 读当前机器人状态
2. `RunFSM(...)` 决定现在该停、归零，还是跑策略
3. `SetCommand(...)` 把目标命令下发到底层

**这仓库里真正重要的 4 层**

1. 数据面：`RobotData`
在 RobotInterface.h。
它就是整个系统共享的“黑板”。
- `q_a_ / q_dot_a_ / tau_a_`：实际状态
- `q_d_ / q_dot_d_ / tau_d_`：期望命令
- `joint_kp_p_ / joint_kd_p_`：底层并联侧增益
- `velEst / gait_a / pos_mode_`：额外控制状态

你可以把它理解成：所有模块都不直接互相调用业务细节，只通过这份状态结构交换信息。

2. 适配层：`RobotInterfaceImpl`
在 RobotInterfaceImpl.cpp。
它解决的是“机器人世界”和“策略世界”之间的翻译问题。

`GetState()` 做三件事：
- 从 Webots 或实机读关节、IMU 等原始数据
- 如果是实机，先做脚踝串并联到等效串联的转换
- 调状态估计，补出浮基姿态和速度

`SetCommand()` 做反方向三件事：
- 先用 PD 把目标位置变成力矩/执行器参考
- 如果是实机，再把等效串联命令转回并联机构命令
- 最后交给仿真或底层驱动

3. 决策层：`RobotFSM`
在 RobotFSMImpl.cpp。
这层只管接口。
状态很少：
- `STOP`
- `ZERO`
- `MLP`

这很重要。现实部署里，RL 只是一个状态，不是整个系统。  
机器人不能上电就直接跑策略，必须先经过安全状态和过渡状态。

4. 策略层：`StateMLP`
在 FSMStateImpl.cpp。
这层才是“把 RL 模型接进来”。

它干的事其实非常标准：
- 读 joystick / navigation 指令
- 做 command ramp，避免指令突变
- 手工拼 obs
- 每 8 个控制周期跑一次 OpenVINO 推理
- 把网络输出重排、缩放、加上默认关节位
- 写回 `robot_data_->q_d_`

**现在最该理解的两个“部署难点”**

1. 为什么要有状态估计
训练环境里，base vel、姿态、接触这些量通常直接可得。  
实机没有这么干净，所以要在 StateEstimate.cpp 里自己估。

这里的逻辑是：
- 用 RBDL 建一个简化刚体模型
- 用动力学和关节力矩反推地面反力：StateEstimate.cpp
- 用接触状态给左右脚不同 trust：StateEstimate.cpp
- 再用卡尔曼滤波融合 IMU 加速度和足端运动学，估计腰部速度/高度：StateEstimate.cpp

从第一性原理讲，这层是在回答一个问题：
“机器人现在到底处在什么状态，才能把它喂给策略？”

2. 为什么要有 `P2S / S2P`
你训练里的动作空间通常假设关节是理想串联铰链。  
但这个机器人脚踝那块不是这么简单，所以实机部署前后都要做机构坐标变换。

这块在 funcSPTrans.cpp。
它做的不是“学习”，而是“机械结构补偿”：
- 实际编码器读数先转成策略理解的等效关节状态
- 策略输出的等效关节命令再转回实际执行器命令

如果不分清这一层，你会误以为“策略输出怎么和电机指令不一样”。其实中间多了一层机构映射。

**怎么读这个仓库，效率最高**

建议顺序：

1. 先读 main.cpp
只看主循环，建立闭环概念。

2. 再读 RobotInterface.h
先搞清 `RobotData` 每个字段代表什么。

3. 再读 RobotFSMImpl.cpp 和 FSMState.h
先接受“RL 只是 MLP 状态”这个事实。

4. 再读 FSMStateImpl.cpp
重点看 obs 是怎么拼的，action 是怎么落地的。  
这是和你训练最直接对应的一层。

5. 最后读 RobotInterfaceImpl.cpp、StateEstimate.cpp、funcSPTrans.cpp
这三块是“实机世界”的脏活，不是先读最划算，但迟早得懂。

**最后一句压缩版定义**

这个仓库的核心不是 C++ 技巧，而是这条链：

`传感器/编码器 -> 状态估计 -> obs 拼接 -> 策略推理 -> 关节目标 -> 机构变换/PD -> 电机`

## 2.分布式训练
布式训练里，你可以先把它理解成“每张卡起一个独立进程，各自跑一份环境和一份网络，然后周期性同步”。

这里有两个最基本的同步动作：

1. 启动同步  
    每个 rank 一开始的模型参数必须一致。常见做法是：
    - 要么所有 rank 用同一个随机种子，各自初始化出完全一样的权重
    - 要么只让 rank0 初始化，然后把参数广播给别的 rank
2. 梯度同步  
    每个 rank 各自算完 loss 和 backward 之后，要把梯度做一次平均，也就是 all_reduce，这样每张卡上的 optimizer.step() 才**等价于“一个更大的 batch”在训练**。
    
    梯度同步本质上就是：
- 每个 rank 都有一份 parameter.grad
- 对每个 grad 做 all_reduce(sum)
- 再除以 world_size
- 然后每个 rank 再各自 optimizer.step()

## 3.ROS

先看**没有 ROS** 时，程序怎么工作。

你写一个 C++ 控制程序，比如 main.cpp：

```cpp
int main() {
  while (true) {
    GetState();
    RunFSM();
    SetCommand();
  }
}
```

这件事本质上只是：

1. 操作系统启动一个进程
2. 进程从 `main()` 开始跑
3. 代码自己决定读什么、算什么、写什么

如果你还想接手柄、相机、导航命令，有两种办法。

第一种，不用 ROS。  
那你就自己处理一切：

- 手柄：自己读 `/dev/input/js0`
- 相机：自己开驱动
- 导航命令：自己开 socket / shared memory / file / 串口
- 多个程序之间通信：自己定义协议

这就是“裸进程 + 自己管通信”。timer += dt_
timer_gait += dt_
policy 调用判断
步态相位推进
zero 动作插值

ROS 干的事情，本质上就是把这部分公共问题统一掉。  
所以我说它“多出来的是一层进程组织和通信框架”。

**什么叫进程组织**

假设你现在不是一个程序，而是 5 个程序：

- 一个读手柄
- 一个做状态估计
- 一个做步态控制
- 一个做可视化
- 一个记录日志

不用 ROS 时，你要自己回答：

- 先启动哪个？
- 哪个挂了怎么办？
- 它们之间怎么找到彼此？
- 配置参数放哪？
- 一次怎么把它们全启动起来？

ROS 就是把这些问题抽象了：

- 每个程序叫 `node`
- 节点之间可以被统一启动
- 可以统一传参数
- 可以统一命名
- 可以统一管理谁和谁连接

这就叫“进程组织”。

比如：
- `ros2 run`：启动一个节点
- `ros2 launch`：按一套配置启动一组节点

注意，ROS 没有改变你的控制算法本身。  
它只是帮你“组织多个进程怎么一起工作”。

**什么叫通信框架**

再看节点之间怎么交换数据。

不用 ROS 时，如果手柄程序要把速度命令给控制程序，你得自己设计：

- 数据格式
- 谁发谁收
- 用 TCP、UDP、共享内存还是管道
- 怎么处理频率、丢包、时间戳

ROS 直接给你一套现成模型：

- `topic`：连续数据流  
  例如 `/cmd_vel`、`/joy`
- `service`：请求-响应  
  例如“切换模式”
- `action`：长任务反馈
- `parameter`：运行参数

所以如果有个 `joy_node`，它做的事情其实很朴素：

- 读取 `/dev/input/js0`
- 把结果包装成 ROS 消息
- 发布到 `/joy`

控制节点不用再自己读设备了，只要订阅 `/joy` 就行。

这就是“通信框架”。

**为什么说 ROS 只是多了一层，而不是替代 C++**

因为你的算法代码还在那里。

比如你这个仓库里真正干活的还是：

- main.cpp
- RobotInterfaceImpl.cpp
- RobotFSMImpl.cpp
- FSMStateImpl.cpp

这些是“业务逻辑”：
- 怎么读状态
- 怎么拼 obs
- 怎么跑 policy
- 怎么发控制命令

ROS 不会替你写这些。  
ROS 只是让这些程序更容易和别的程序协作。

所以更准确地说：

- **C++/Python**：解决“程序内部怎么算”
- **ROS**：解决“多个程序怎么连起来一起跑”

**用一个很具体的类比**

把机器人系统想成一个公司。

没有 ROS：
- 每个人自己打电话、发微信、拉群、约时间
- 也能干活，但通信和协作全靠自己维护

有 ROS：
- 公司提供统一办公系统
- 大家有统一通讯录、会议室、流程、公告板

但真正做设计、做代码、做实验的人还是原来那些人。  
办公系统没有替他们工作。

ROS 对机器人程序就是这个“统一办公系统”。

**落到当前这个项目**

你现在 Webots 这条链，基本没怎么用 ROS。

因为是：

- Webots 直接启动 controller
- controller 从 main.cpp 进入
- 然后自己跑控制循环

这是一种“单主程序闭环”。

而如果你启用 `USE_ROS_JOY`，事情就变成：

- 有 ROS 节点负责发布 joystick / navigation 命令
- 你的控制程序去订阅这些消息
- 这时 ROS 开始参与“进程组织和通信”

但即使这时，控制逻辑本体仍然是你的 C++ 代码，不是 ROS 替你控制机器人。

**最短总结**

“ROS 多出来的只是一层进程组织和通信框架”的意思就是：

- 它不替你写控制算法
- 它不替你决定 policy 怎么算
- 它主要负责：
  - 多个程序怎么启动
  - 多个程序怎么互相发消息
  - 参数、命名、服务怎么统一管理

如果你愿意，我下一条可以直接用你这个仓库里的两条实际路径画给你看：

1. `Webots 模式` 到底是谁启动谁  
2. `USE_ROS_JOY 模式` 到底 ROS 多插进了哪一层

## 4.Sim2Real
**第一性原理**
真机部署本质上不是“跑一个 policy”，而是每 2.5ms 做一次闭环控制：

1. 读真机状态：电机编码器、速度、电流、IMU、手柄或导航速度。
2. 坐标/单位转换：把真机 CAN 电机 ID、零位、方向、IMU 欧拉角等转换成训练/策略期望的状态。
3. 运行状态估计：从 IMU 和关节状态得到 base 姿态、速度等 policy observation 需要的信息。
4. 运行 FSM：决定当前是 `STOP`、`ZERO` 还是 `MLP`，以及是否让 RL policy 接管。
5. 推理 policy：把 observation 输入模型，得到目标关节动作。
6. 输出低层控制：把 policy 输出转换为目标关节位置/速度/力矩，再通过 PD 参数发到电机驱动。
7. 安全退出：如果状态异常、急停、禁用关节，就停止发力或退出。

在这个仓库里，这个闭环主线在 RLControlPlugin.cpp。

**代码分层**
RLControlPlugin.hpp 是 ROS2 插件入口。它负责订阅和发布，不直接代表策略本身。

它订阅这些真机输入：

`/leg/status`：腿部电机状态。

`/arm/status`：手臂电机状态。

`/imu/status`：IMU 状态。

`/sbus_data/event`：遥控器输入。

`/hric/robot/cmd_vel`：导航速度输入。

它发布这些控制输出：

`/leg/cmd_ctrl`：腿部电机命令。

`/arm/cmd_ctrl`：手臂电机命令。

`/waist/cmd_pos`：腰部回零命令。

`/hric/motion/status`：运控状态反馈。

真正的策略/FSM接口来自这几个头文件：

RobotInterface.h 定义 `RobotInterface`，它是“状态估计 + policy 输入输出转换”的抽象接口。

RobotFSM.h 定义 `RobotFSM`，它决定当前状态机行为。

FSMState.h 定义状态枚举：`STOP`、`ZERO`、`MLP`。

但这些类的实现不在源码里，而是链接进 libx_humanoid_rl_sdk.so。所以如果要对齐我们之前在 `tg2_lab_sdk` 里改过的 half-walk policy 逻辑，核心实现要么已经被重新编译进这个 `.so`，要么你需要拿到 SDK 源码重新生成 `.so`。

**主循环数据流**
主循环一开始创建 FSM 和接口：

RLControlPlugin.cpp

```cpp
RobotFSM *robot_fsm = get_robot_FSM(robot_data);
RobotInterface *robot_interface = get_robot_interface();
robot_interface->Init();
```

这说明 `rl_control` 自己不 new 具体类，而是从动态库里拿工厂函数。第一性原理上，这是一个“ROS2 外壳调用 SDK 控制核心”的架构。

每个控制周期做这些事：

读 ROS 队列里的最新关节和 IMU。

把 32 个全身电机数组压缩成 20 个控制关节：12 腿 + 4 左臂 + 4 右臂。

用 `zero_pos`、`motor_dir`、`zero_offset` 把真机编码器角度变成控制内部角度。

把结果写入 `robot_data.q_a_ / q_dot_a_ / tau_a_`。

把 IMU 写入 `robot_interface->imu_data_`。

调用 `robot_interface->GetState(timeFSM, robot_data)`。

调用 `robot_fsm->RunFSM(flag_)`。

调用 `robot_interface->SetCommand(robot_data)`。

从 `robot_data.q_d_ / q_dot_d_ / tau_d_` 取出目标命令。

再转换回真机电机坐标，发布 `/leg/cmd_ctrl` 和 `/arm/cmd_ctrl`。

关键点是：`robot_data` 是外壳和 SDK 之间的共享数据合同。真机观测写进去，SDK 读它并生成命令，再写回同一个结构。

**真机最危险的地方**
第一类风险是**坐标和关节顺序**。`bodyIdMap` 定义了全身 32 电机的物理 ID 到数组 index 的映射：bodyIdMap.h。主循环又把 32 维映射到 20 维控制关节。如果这里和训练资产、Webots、SDK policy 顺序不一致，policy 会把“髋关节动作”发给“膝关节”或反号，真机会非常危险。

第二类风险是**零位和方向**。配置文件固定在 `/home/ubuntu/data/param/rl_control_new.txt`，里面读 `zero_pos_0...`、`xsense_data_roll`、`xsense_data_pitch`、速度 offset 等。零位或 IMU offset 错了，policy 看到的机器人姿态就是假的。

第三类风险是**时基**。这里控制周期硬编码为 2.5ms：

RLControlPlugin.cpp

RLControlPlugin.cpp

FSMState.h

外壳每 2.5ms 跑一次，但 policy 可能不是每个周期都推理，`FSMState` 里有 `freq_ratio_ = 4`，这通常意味着 policy 控制周期是 `2.5ms * 4 = 10ms`，对应 100Hz。这个必须和训练中的 `sim.dt`、`decimation` 对齐。

第四类风险是 **PD 参数**。这里真机 PD 是硬编码的：

RLControlPlugin.cpp

policy 输出不是直接电流控制，而是经由目标位置/速度/力矩加 PD 下发。如果训练里的 actuator 模型、stiffness、damping 和这里差很多，sim2real 会变差。

第五类风险是 failfast 目前不够一致。初始化时如果十秒内所有关节位置还是 0，会直接退出，这是好的：RLControlPlugin.cpp。但运行时对关节跳变的处理是“打印并沿用上一帧”，这是真机上常见的保护，但对调试 sim2real 问题可能会掩盖错误。

**和之前 Webots 部署的关系**
`tg2_lab_sdk` 更像完整 SDK 源码侧，能看到 `FSMStateImpl.cpp`、policy observation、OpenVINO 推理、动作后处理等核心逻辑。

`rl_control` 更像真机 ROS2 接入侧，只暴露了 SDK 的头文件和 `.so`。所以它的部署地位是：

`rl_control`：真机消息桥 + 实时循环 + 电机发布。

`x_humanoid_rl_sdk.so`：状态估计 + FSM + policy 推理 + 动作生成。

`python_scripts/traced/*.xml/bin`：OpenVINO policy 模型文件。

如果要把 half-walk 真机部署打通，原则上要确认三件事：

1. `libx_humanoid_rl_sdk.so` 内部加载的是这次 half-walk 的 `lite_half_walk_policy.xml/bin`，不是旧的 `mwr_tg2_lite_0807.xml/bin`。
2. SDK 内部 observation/action 逻辑和 `tk_lab` 训练环境完全一致，包括关节顺序、历史帧、command scale、default pose、action scale、decimation。
3. `rl_control` 外壳给 SDK 的真机状态和训练定义一致，包括 IMU 轴、欧拉角顺序、角速度单位、关节角单位、零位、正方向、时基。

## 5.串并联
**问题1**：反解这一步的原因我还是不懂，从串联到并联的映射我倒是懂了，映射出来三个量目标关节角、目标关节角速度、目标转矩这三者地位不是一样的吗，最终不就是要驱动现在的关节角到目标关节角来满足机器人policy动作吗，转矩是多少无所谓吧？
**答**：policy 学到的是“我给一个目标角偏差，PD 会产生某个力矩响应”，**不是**“必须精确追踪这个角度轨迹”。
反解不是为了让“真实电机角度轨迹”和训练里的“等效踝关节角度轨迹”逐点一致，而是为了让真实并联机构对 policy 动作的**力学响应**尽量接近训练/仿真里的响应。

**问题2**：但是我们不是已经求出来目标转矩了吗，为什么不直接用呢，而是还要用关节角？
**答**：反解的目标就是让真机并联踝最终产生的力矩响应，尽量接近训练 / sim2sim 里等效串联踝的力矩响应。
关键问题是：我们虽然已经算出了 `目标转矩 tor_d_p`，但真机底层接口未必允许我们“直接、可靠地让电机输出这个转矩”。当前代码走的是电机 PD 位置接口，而不是纯力矩接口。

底层实际控制大概率是这种形式：

```text
tau_motor =
  Kp_motor * (q_cmd - q_actual)
+ Kd_motor * (qd_cmd - qd_actual)
+ tau_ff
```

也就是说，电机真正输出的力矩不是你上层说一个 `tor_d_p` 就完全等于 `tor_d_p`。它由 `pos / spd / kp / kd / tor` 一起决定，尤其由位置误差和速度误差决定。

如果真机接口是理想力矩控制，那当然可以直接用：

```text
motor.tor = tor_d_p
motor.kp = 0
motor.kd = 0
```

但很多机器人电机控制链路不适合这么做，原因通常是：

1. 底层电机/驱动器主要设计为位置/速度/阻抗接口，力矩项可能只是前馈补偿，不是高带宽闭环力矩控制。
2. 直接力矩控制对通信延迟、摩擦、模型误差、限流、保护策略更敏感，真机上更容易抖、软、失控或者保护。
3. 关节位置 PD 是更稳定的“安全外壳”：即使上层算的力矩不准，电机仍围绕当前位置附近输出受限阻抗，而不是完全开放力矩。
4. 当前 `rl_control` 发布的消息同时带 `pos/spd/tor/kp/kd`，但 SDK 在并联踝这里最后把 `tau_d_` 置零，说明这个部署链路选择的是“把目标力矩编码成位置偏差”，而不是直接发力矩前馈。

代码里能看到这个选择：RobotInterfaceImpl.cpp:160 先用 `tor_d_p` 反解 `q_d_p`：

```cpp
q_d_p(i) = (tor_d_p(i) - Kd_p * (qdot_d_p(i) - qdot_a_p(i))) / Kp_p + q_a_p(i);
```

然后把并联踝的直接力矩命令清零：

```cpp
robot_data.tau_d_.segment(10, 2).setZero();
robot_data.tau_d_.segment(16, 2).setZero();
```

所以最终发给真机的不是：

```text
请直接输出 tor_d_p
```

而是：

```text
请用你的电机 PD，追踪我反解出来的 q_d_p；
这样你内部 PD 算出来的力矩约等于 tor_d_p。
```

为什么还要用关节角？因为当前底层的执行接口就是“位置型阻抗控制”。上层想要一个力矩，但底层擅长根据“位置误差”产生力矩。于是上层把力矩转换成位置误差：

```text
tau = Kp * position_error + Kd * velocity_error
```

反解：

```text
position_error = (tau - Kd * velocity_error) / Kp
```

再得到：

```text
q_cmd = q_actual + position_error
```

这就是“用关节角承载力矩意图”。

一个直观例子：

假设某个并联电机当前角度是 `1.00 rad`，底层 `Kp = 100`，速度项先忽略。上层希望它输出 `10 Nm`。

如果直接用位置控制，你应该发：

```text
q_cmd = 1.00 + 10 / 100 = 1.10 rad
```

电机看到 `0.10 rad` 位置误差，就会打出：

```text
100 * 0.10 = 10 Nm
```

这里 `1.10 rad` 不一定是你真的希望它长期到达的轨迹点，它只是为了让电机此刻产生 `10 Nm` 的“虚拟位置命令”。

所以这不是“目标转矩没用”，恰恰相反：`tor_d_p` 是核心；`q_d_p` 只是把这个目标转矩送进位置 PD 底层的一种实现手段。

**所以，训练和真机，目标力矩一样，角度映射不一样，那么PD参数也会不一样。**

**gait_feet_frc_perio、gait_feet_spd_perio、gait_feet_frc_support_perio以及他们为什么在yaw only、y only、低速、小步、height/roll/pitch 混合命令表现不如feet_contact_alternation、feet_air_time_positive_biped、gait_cycle_reward、contact_no_slip？**
问题在于这三个 reward 都假设：**机器人应该按固定相位表走路**。这个假设在标准前进大步里还行，但在 yaw-only、y-only、低速、小步、height/roll/pitch 混合时会变差。因为这些动作需要 policy 自己改变步态节奏、支撑时间、双支撑比例、脚落点和步幅，而固定 gait phase 会持续告诉它“这个时间左脚必须 swing、那个时间右脚必须 support”。这会把“脚该怎么接触地面”变成硬模板，而不是根据 command 和身体状态自适应。yaw-only 最典型。原地转不一定需要和前进一样的大幅交替摆腿，它更需要小步、绕身体轴心、少 xy drift、双脚 midpoint 稳定。如果固定相位强推 swing/support，policy 容易学成绕某只脚转，或者用一只脚大步、一只脚小步来满足相位接触要求。

### 6.2 pro half walk move operate

move operate 是在 pro half walk 上把 command 从 3D velocity 扩成 6D：`[lin_vel_x, lin_vel_y, ang_vel_z, roll, pitch, height]`。核心不是多几个 reward，而是重新定义 standing、默认姿态和 height 先验。

2026-05-23 这次修改的定位：它不是把已经验证过的 `pro_half_walk_yaw_mix` 那套复杂 mode-aware reward 和权重机制整体搬进 move operate，而是采用 `c9e30b5` 的 gait-free 思路来修 yaw、y-only command 下的运动表现。核心是：去掉 actor obs 里的 gait phase，禁用旧 clock-gait 三件套，用接触/腾空/周期估计/防滑这类不依赖显式相位的 reward 替代；同时把 `heading_command=False`，让 yaw command 直接作为 `ang_vel_z` 采样，而不是由 heading controller 间接生成。

| reward 名称 | 典型权重 | 说明 |
| --- | ---: | --- |
| `track_lin_vel_xy_exp` | `4.0` | 继承 gait-free 配置，`std=0.25`。比旧 `0.5` 更严格，用来加强 x/y command，尤其是 y-only 或 yaw-only 下对非目标方向漂移的压制。 |
| `track_ang_vel_z_exp` | `2.0` | 继承 gait-free 配置，`std=0.25`。配合 `heading_command=False`，让 yaw rate command 的监督更直接。 |
| `low_speed_command_response_bonus` | `-0.5` | 防止低速 x/y/yaw command 被 policy 当成近似 zero command 忽略。对 move operate 很关键，因为 6D command 已经稀释了每个 velocity 子模式的采样密度。 |
| `track_roll_pitch_exp` | `8.0` | 跟踪 roll/pitch command。误差越小指数奖励越高。 |
| `track_height_exp` | `8.0` | 跟踪 base height command。`std=0.07`，比早期 `0.3` 更窄，避免默认高度也拿高分。 |
| `base_height_error_l1` | `-5.0` | height 远离目标时提供线性拉力，弥补指数 reward 远处梯度弱的问题。 |

### 6.3 dex move operate

dex 和 pro 最大的构型差异：**dex 有一个 3-DOF 可动腰（`waist_yaw/roll/pitch`），pro 没有（腰是冻结上身的一部分）**。所以 dex 的 move operate 不照搬 pro“靠腿把整个刚性上身扳过去”的做法，而是用腰来做“操作”姿态。

设计（Option B，gait-free 与 gait-clock 两个 worktree 各一份）：

- **控制 15-DOF = 12 腿 + 3 腰**；**双臂冻结为 nuisance**（随机固定目标，部署时由单独的上肢栈控制，沿用 pro 半身那套上身扰动建模思路）。动作排布 `腿(0–11) + 腰(12–14)`，前 12 维与 pro 腿序一致，所以 `hip_roll_action / ankle_action` 这些按动作 index 的 reward 直接复用。
- **roll/pitch → 上半身（腰）**：新增 `track_roll_pitch_body_exp`，跟踪 `waist_pitch_link`（躯干）的世界姿态而不是骨盆。骨盆靠 `body_orientation_l2 + flat_orientation_l2`（常开、不随命令门控）保持竖直当稳定底座；**height → 骨盆**（靠腿下蹲），走路 / 转向靠腿。腿做“稳定底座 + 移动 + 下蹲”、腰做“操作倾斜”，分工干净，也规避了 pro 上“整机大幅前倾难平衡”和“下蹲漏 roll”。
- **腰正则**：`waist_yaw` 常向中立位正则（无对应命令维度，可微辅助转向）；`waist_roll/pitch` 用新增的 `joint_deviation_l1_rp_neutral`，只在不给 roll/pitch 命令时拉回中立——注意它只看 roll/pitch neutral、不看 height，所以纯下蹲（给 height、rp 中立）时也保持腰中立，正面规避 pro 的下蹲漏 roll。再加一个 `waist_joint_velocity` 平滑。
- **复用 pro 的 move operate 机制**：6D `UniformVelocityRPHCommand` + command mode（neutral_stand / in_place_pose）+ walk-first pose 课程 + speed-cap 速度门控，全部直接复用；`neutral_height=0.97`（dex 站立高度，pro 是 0.95）；范围 `roll ±0.35 / pitch -0.4~0.85 / height 0.62~0.97`（按 dex 腰关节限位 `waist_roll ±0.436 / waist_pitch -0.52~0.96` 留余量）。

- **终止**：接触 + 超时 + 骨盆倾倒绝对保护（`projected_gravity_b[:,2] > -0.3`）。dex 方案里骨盆始终竖直，不需要 pro 那套“相对命令”的终止。

单帧 actor 维度是 `45`。policy 只看下半身关节，arms/waist 不进入 action，也不作为 actor joint obs 出现。当前 gait-free 版本已经去掉 `sin(gait_phase) / cos(gait_phase) / phase_ratio` 这 6 维；这样 policy 不再被显式 clock 驱动，yaw-only、y-only、normal command 更依赖 command 和状态本身来形成步态。

旧版 actor obs 中已经删除的 gait phase 项：

- ~~`sin(gait_phase)(2)`~~
- ~~`cos(gait_phase)(2)`~~
- ~~`phase_ratio(2)`~~
- ~~单帧 actor 维度 `51`~~

单帧 critic obs：

```text

root_lin_vel_b(3)
feet_contact(2)
upper_body_nuisance_obs(15)
```

`upper_body_nuisance_obs` 包括：`arm_target_offset(8)`、`waist_target_offset(1)`、`pelvis_mass_delta(1)`、`pelvis_com_offset(3)`、`latest_pelvis_push(2)`。critic 看到这些扰动，是为了让 value function 更容易解释环境随机性；actor 看不到，避免部署时依赖不可观测扰动。

旧版 critic 维度记录：

- ~~`actor_obs(51) + root_lin_vel_b(3) + feet_contact(2) + upper_body_nuisance_obs(15) = 71`~~

yaw mix 不改变基础 obs 结构。它只改变 command 采样和 reward：yaw-only 样本仍是 3D command，只是强制 `[0, 0, yaw]`；y-only 样本强制 `[0, y, 0]`。注意：去掉 gait phase 后，旧的 phase-gait reward 不应继续作为主步态约束，否则 reward 目标依赖一个 actor 看不到的内部 clock。

### 7.2 pro half walk move operate obs

move operate 继承 pro half walk 的 obs 拼接方式，主要差异是 command 从 3D 变成 6D：

```text

```

旧的含 gait phase 版本记录：

- ~~pro half walk 单帧 actor 维度 `51`~~
- ~~move operate 单帧 actor 维度 `54`，也就是 `51 + roll/pitch/height(3)`~~
- ~~move operate critic 维度 `74`，也就是 `71 + roll/pitch/height(3)`~~

这条旧记录仍然说明了一件事：`roll/pitch/height` 必须进入 actor obs，policy 才能学会姿态和高度操作。只是当前 gait-free 版本先删除了 gait phase 6 维，所以最终维度变成下面的 `48/68`。

gait-free 版本以后，move operate 的单帧 actor 维度是 `48`：

```text

projected_gravity_b(3)

leg_joint_pos_error(12)
leg_joint_vel(12)
last_action(12)
```

critic 维度是 `68`：`actor_obs(48) + root_lin_vel_b(3) + feet_contact(2) + upper_body_nuisance_obs(15)`。这次 obs 变化会导致旧 checkpoint 的 actor 输入维度不兼容，不能直接 resume。

height scanner 当前主要服务 reward：`track_base_height_exp` 和 `base_height_error_l1` 用它估计 base-to-ground height。它不是额外拼进 actor obs 的 height scan map；actor 看到的是 height command，不直接看到地形扫描点。

standing 语义也不同：6D command 下 standing 只清 `lin_vel_x/lin_vel_y/ang_vel_z`，保留 `roll/pitch/height`。否则 standing 会把 height 清到 `0.0`，直接污染训练。

### 7.3 部署 obs 对齐原则

部署侧必须和训练侧保持以下内容一致：关节顺序、默认关节位、action scale、command scale、obs 顺序、history 帧数、policy 推理频率。half walk 的核心边界是：policy 输出只对应 12 个腿部 action，上身和 waist 由默认 PD 目标控制。

zero2mlp 抖动和 obs 有关。旧版本 actor 看到 gait phase，容易输出相位相关 residual；gait-free 版本去掉了 gait phase，但 actor 仍然看到 last action、关节速度和 IMU，所以 MLP 仍可能输出小 residual。部署切换时如果 action history / 当前关节状态没有和训练中的 MLP 闭环状态对齐，第一帧 target 仍可能不连续。

### 7.4 dex move operate obs

dex 控 14 个关节（12 腿 + 2 腰 `waist_roll/pitch`，`waist_yaw` 冻结），所以 obs 维度和 pro（控 12 腿）不同。gait-free 版本单帧 actor 维度是 `54`：


## 8.工作总结

> [!NOTE] 2.0 pro half walk 工作总结：
> **核心改动**
> 本质是把原来的全身 walk 任务拆出“半身站走”任务：policy 只控制下半身 12 个关节，上半身和 waist 不进 action，而是用默认 PD 目标维持，并在训练里把上半身/骨盆当扰动源建模。
> 
> 具体来说：
> 
> - 新增 `lite_half_walk`、`pro_half_walk`、`pro_half_walk_yaw_mix` 三类任务注册。
> - 新增 lite/pro 半身 env/cfg。
> - `num_actions` 从全身动作变成下半身 12 维，`leg_joint_ids = left_leg_ids + right_leg_ids`，上半身 `arm_joint_ids` 和 `waist_joint_ids` 单独维护。
> - step 里只把 policy action 写到腿部关节；waist 和 arms 写入目标默认 PD 位置。
> - actor obs 只包含下半身 joint pos/vel/action、command、重力、角速度；critic 额外看到 root lin vel、feet contact 和 upper-body nuisance。
> - ~~旧版 actor obs 还包含 `sin(gait_phase)(2) / cos(gait_phase)(2) / phase_ratio(2)`，单帧 actor 维度 `51`。~~
> - 2026-05-23 gait-free 版本去掉 gait phase 相关 6 维 obs，单帧 actor 维度变为 `45`；critic 从 ~~`71`~~ 变为 `65`。
> 
> **训练建模**
> 我们不是简单“锁死上身”，而是让策略在上身扰动下学会下半身稳定走：
> 
> - arm target 随机偏移、waist target 随机偏移、pelvis mass 随机、pelvis COM 随机、pelvis push 随机。
> - 禁用了原始 `add_base_mass`、`push_robot`、`randomize_rigid_body_com`，在其基础上改为半身任务自己的 pelvis 级扰动，避免重复扰动和语义不清。
> - 保留/调整了外力、PD gain、joint params、material 等 domain randomization。
> - agent 改成 AMPPPO，AMP 只取适合半身策略的 observation index，并使用半身专用 symmetry。
> 
> **Reward 和 yaw mix**
> 基础 `pro_half_walk` 上，我们做了半身站走 reward 改造：去掉 arms deviation，stand still 覆盖 lower body + waist，增加脚太近、横移脚前后对齐等约束。后续主要问题集中在 yaw-only 和 y-only command 的样本分配、reward 互相拉扯，以及部署时 zero2mlp/standing 的小抖动。
> 
> - 原始 command 采样几乎没有纯 yaw-only 样本，导致原地 yaw 时有明显 x/y 漂移，所以新增 `pro_half_walk_yaw_mix`。
> - 早期 `yaw_only_ratio=0.5` 能压 yaw 漂移，但会挤占正常 locomotion 和 y-only 学习；后来改成 `yaw_only_ratio=0.2`、`y_only_ratio=0.2`，剩余样本仍走普通 command。
> - yaw-only 只给 `ang_vel_z`，强制 `lin_vel_x/y=0`；y-only 只给 `lin_vel_y`，强制 `lin_vel_x/yaw=0`。
> - yaw-only 增加 `yaw_only_lin_vel_x/y_l2`、`yaw_only_feet_midpoint_xy_l2`、`yaw_only_feet_cycle_path_symmetry_l1`，用于压 x/y drift、避免绕某一只脚转、约束一个 cycle 内左右脚平面路径接近。
> - `yaw_only_feet_stance_width` 后来删除；它会和实际转身站姿/脚间距策略互相拉扯，不如保留通用脚距约束。
> - 为 y-only 新增 `y_only_lin_vel_x_l2` 和 `y_only_feet_air_time_positive_biped`，解决横移只训 yaw 后被遗忘的问题。
> - `feet_sagittal_align_lateral` 改回 `require_double_support=True`，避免横移 swing phase 被脚前后对齐 reward 过度压制。
> - `yaw_only_gait_*` 不继续加大。定量看它们已经不弱，再加会把问题推向拖脚/原地搓脚。
> - `yaw_only_feet_midpoint_xy_l2: -25 -> -40`、`yaw_only_feet_cycle_path_symmetry_l1: -2 -> -3` 后，平面轴心和路径对称确实改善，但暴露出内侧脚低抬的问题。
> - 为内侧脚低抬新增 `yaw_only_swing_clearance_l1`：只在 yaw-only 下，用 gait swing phase 判断该抬哪只脚，用 `swing_foot_z - opposite_ankle_z` 约束相对 clearance，目标先保守取 `0.04m`，权重 `-2.0`。
> - ~~旧基础 `pro_half_walk` 使用 `gait_feet_frc_perio / gait_feet_spd_perio / gait_feet_frc_support_perio` 三个 clock-gait reward，并把 gait phase 显式放进 actor obs。~~
> - 2026-05-23 之后，`pro_half_walk` / `pro_half_walk_move_operate` 采用另一条更简单的修复路线：不把 yaw-mix 的复杂 mode-aware reward 和权重机制整体搬进 move operate，而是采用 gait-free 范式，禁用旧 `gait_feet_*` 三件套，改用 `feet_contact_alternation / feet_air_time_positive_biped / gait_cycle_reward_adaptive / contact_no_slip`，并移除 actor obs 的 gait phase。
> 
> yaw-only command 下期望的是：
> 
> - `cmd = [0, 0, yaw_rate]`。
> - base yaw_rate 跟踪 cmd。
> - base x/y 速度尽量为 0。
> - 左右脚中点围绕 base，不绕某一只脚转。
> - 一个 cycle 内左右脚累计平面运动量接近。
> - swing 脚不能贴地拖，尤其不能出现“左转左脚低抬、右转右脚低抬”的内侧脚低 clearance 解。
> 
> **训练经验**
> 最近比较稳定的一版是 `2026-05-20_10-51-21_10ms_sym100_1gpu_8192env`。可视化上 `24500` 和 `25000` checkpoint 最好，再继续训整体行为会变差；后续同类 yaw-mix resume 优先训练到 `25000` 左右截断，而不是盲目训满后段。
> 
> 对后段变差的判断：TensorBoard 总 reward 不一定崩，甚至可能继续涨，但局部行为会在 yaw/y/stand 之间重新折中，表现为转身或站立细节变差。因此这类任务不能只看总 reward，必须固定检查 yaw-only、y-only、normal command 和 zero command 的可视化。
> 
> **standing / zero2mlp 抖动**
> 当前 standing 和 zero2mlp 仍会有小抖动，主要是结构性原因：
> 
> - `command=0` 时 policy 不是 hard-coded 静止控制器。旧版本还会吃 `gait_phase`，容易输出相位相关的小 residual；gait-free 版本去掉 gait phase 后，仍然会受上一帧 action、关节速度、IMU 等 obs 影响。
> - `stand_still_*` 只是 soft reward，不是强制 action 为 0；它能压小动作，但不能保证完全静止。
> - `action_rate_l2` 只约束 MLP rollout 内连续 action，不约束 ZERO 状态切到 MLP 的第一帧 target 跳变。
> - zero2mlp 是部署侧分布外切换：外部 ZERO/FSM 站立状态切入 MLP 时，action history、实际关节状态可能和训练中 MLP 闭环暖机后的状态不一致；旧含 gait phase 版本还多一个 phase 对齐问题。
> - 如果要继续修，部署侧更直接的是 zero2mlp 做短时间 target/action blend 或初始化 MLP action history；训练侧则加强 zero-command 下 action/feet/body velocity 约束。
> 
> **配套基础设施**
> 还修了之前的训练基础设施：
> 
> - 多卡训练 device/seed 处理更明确。
> - 新增 `_distributed.py`，把多卡 gradient reduce 从拼大 tensor 改成逐参数 all-reduce，并让 AMP discriminator 也同步梯度。
> - AMP runner 支持 `amp_observation_indices`，也就是半身 AMP 只喂下半身相关 expert obs。
> - 新增 `symmetrylite_half_walk.py` 和 `symmetrypro_half_walk.py`，因为半身 obs/action 维度已经不同于原全身任务，不能继续复用原来的镜像函数。
> 
> 一句话总结：这段差异把原来的全身 walk 代码扩展出 lite/pro 半身站走任务，并进一步加了 pro yaw-mix 实验任务；核心工程边界是 **RL policy 只管下半身，上身/waist 默认 PD + 扰动建模，critic 看扰动，AMP/symmetry/reward/多卡训练都为这个半身任务重新适配**。最新经验是 yaw/y/normal/zero 四种模式不能只靠总 reward 判断，需要用 checkpoint 可视化截断训练，当前优先采用 `25000` 左右的 checkpoint。
> **command 死区的问题**：mlp相对线性，当 rel_stand_env = 0.4这个比例的时候，mlp policy 偏向于站立，这种策略在训练的时候能拿到很多奖励，所以有较大死区；把 rel_stand_env 调到 0.2 死区变小。如果是 MOE 就不会有这个问题。

> [!NOTE] 2.0 pro half walk move operate 工作总结：
> **目标**
> 在已经成功的 `pro_half_walk` 基础上引入 move operate 的 6D command：`lin_vel_x / lin_vel_y / ang_vel_z / roll / pitch / height`。工程原则不是在原生全身 `move_operate` 上重做半身，而是以半身站走为基底，选择性 port 原生 move operate 的 command / reward / play 逻辑。
>
> **主要问题与对策**
> 1. `rel_standing_envs` 语义错误。
>    原生 `UniformVelocityCommand` 只有 3D velocity command，所以 standing 清零全部 command 没问题；扩展成 6D 后如果仍清零全部维度，就会把 `height` 清成 `0.0`，污染训练。后来修正为：standing 只清 `x/y/yaw`，保留 `roll/pitch/height`。
>
> 2. `stand_still_exp` 和 6D command 冲突。
>    旧 `stand_still_exp` 只看 `x/y/yaw≈0`，不看 RPH。这样在“原地做 roll/pitch/height”时也会奖励腿回默认站姿，和姿态/高度跟踪冲突。后来改为 `stand_still_exp_rph_neutral`：只有零速度且 `roll≈0, pitch≈0, height≈0.95` 时才奖励默认腿姿态。
>
> 3. height command 被 policy 忽略。
>    现象是 play 里无论 `height` 填多少，机器人 base height 都接近 `0.95`。`track_height_exp(std=0.3)` 太宽，默认高度不动也能拿到很高平均分；**收紧到 `std=0.07` 后，TensorBoard 才暴露出 height 没真正跟踪**。
>
> 4. 只收紧指数 reward 仍不够。
>    `std=0.07` 让近目标精修更强，但远离目标时指数 reward 梯度很弱。于是新增 **`base_height_error_l1` 作为远目标拉力**，并逐步把权重加到 `-5.0`。这能把问题暴露得更清楚：policy 仍然倾向保持 nominal height。
>
> 5. 真正压住 height 的是 nominal-height 先验。
>    低 height 必须持续弯髋、弯膝、改踝，会和 AMP、gait、energy、action rate、接触稳定性、默认腿姿态等约束冲突。后续改动是：**`joint_deviation_legs` 加 height gate，`cmd_h < 0.90` 时关闭腿部默认位惩罚；同时把 `amp_reward_coef` 从 `0.3` 降到 `0.1`**，减弱 command-agnostic nominal walk 先验。
>
> 6. yaw / y-only command 下运动表现不稳。
>    ~~一种做法是把 `pro_half_walk_yaw_mix` 中已经验证过的复杂 mode-aware reward、yaw-only/y-only 采样比例和多组权重机制搬到 move operate。~~ 这会把 6D command 任务变得更复杂，并且会继续引入“不同 mode 的 reward 互相拉扯”的问题，可以解决但是复杂。这次采用的是另一条路线：gait-free 思想**。具体是：~~`heading_command=True` 由 heading controller 间接生成 yaw command~~ -> `heading_command=False`，yaw command 直接作为 `ang_vel_z`；`track_lin_vel_xy_exp/std` 和 `track_ang_vel_z_exp/std` 从 ~~`0.5`~~ 收紧到 `0.25`；加入 `low_speed_command_response_bonus` 压低速 command dead-zone；去掉 actor obs 中的 ~~gait phase~~；禁用旧 ~~`gait_feet_frc_perio / gait_feet_spd_perio / gait_feet_frc_support_perio`~~，改用 `feet_contact_alternation / feet_air_time_positive_biped / gait_cycle_reward_adaptive / contact_no_slip` 这些不依赖显式相位的 reward。
>
> 7. RPH 后 zero command 不能再简单等价于 stand still。
>    之前的任务大致是 moving / stand still 二分类；有了 RPH 后更准确是速度任务和姿态任务的组合：`vel_active + pose_neutral`、`vel_active + pose_active`、`vel_zero + pose_neutral`、`vel_zero + pose_active`。其中 `vel_zero + pose_active` 不是默认站立，而是**原地完成 roll/pitch/height**，不能用旧 `stand_still_exp` 强拉默认腿姿态；否则会和 height/roll/pitch 跟踪冲突。后续改成显式 command mode：`neutral_stand_ratio=0.15` 固定 `x/y/yaw=0, roll=0, pitch=0, height=0.95`；`in_place_pose_ratio=0.25` 固定 `x/y/yaw=0` 但随机 RPH；剩余 `0.60` 做 locomotion。reward 上新增 zero-velocity 约束：`zero_velocity_base_xy_vel_l2 / zero_velocity_yaw_vel_l2 / zero_velocity_feet_xy_vel_l1`，用于约束原地 RPH 时不要后退、不要自转、不要踏步搓脚。
>
> 8. zero-velocity 只能压速度，不能定义 in-place RPH 的足端位置。
>    5.27 的改动后，neutral stand 基本不再晃，但 `x/y/yaw=0` 单独给 RPH 时，机器人仍会通过挪脚调整支撑来完成姿态。这里的问题不是“零命令站不住”，而是 in-place pose control 缺少脚固定语义。后续新增 `in_place_feet_anchor_xy_l1`：进入 in-place command 周期时记录两脚 world xy，周期内惩罚脚相对 anchor 漂移。这样 `vel_zero + pose_active` 的目标就变成“脚固定，身体做 RPH/height”，而不是“靠挪脚重新找支撑”。
>
> 9. gait-free 不是只靠接触交替，还要补方向几何和落地质量。
>    5.27 的可视化显示：前进是低频高抬腿重砸脚，后退是高频搓地，侧移是一只脚踩到另一只脚。对应修法是：侧移重新启用 `feet_sagittal_align_lateral`，并新增/启用 `feet_lateral_distance_lateral`；前进新增 `touchdown_foot_vel_z_l2` 压落地瞬间向下速度；后退小幅加强 `contact_no_slip`，并把 `gait_cycle_reward_event` 的权重和 early/late touchdown penalty 加大。这个方向比继续盲目加 `feet_air_time` 更稳，因为前进已经会抬脚，缺的是落地质量和足端几何。
>
> 10. 6.4 补 moving-only 的非饱和速度误差项。
>    `track_lin_vel_xy_exp / track_ang_vel_z_exp` 是近目标指数项，完全不会走时误差大、梯度弱，所以新增 `moving_lin_vel_xy_error_l1 / moving_yaw_vel_error_l1 / moving_directional_response_l1`，只在 moving command 下给速度远目标拉力。初始 `-4 / -2 / -3` 在 `2026-06-04_10-21-30` 里证明方向有效：按 alive time 归一化后速度响应更强；但 episode length 和 terrain 明显变差，说明它在步态事件还没成形时压过了 gait event / air-time / touchdown。后续两个 worktree 都降成 `-1.5 / -0.75 / -1.0`，把这三个项定位成辅助拉力，而不是早期主目标。
>
> **得到的效果**
> - ~~旧 move operate actor 单帧维度是 `54`，critic 是 `74`。~~ gait-free 后 actor 单帧维度是 `48`，critic 是 `68`。
> - roll/pitch 能学到，说明 RPH command 并不是整体不可用；height 难点来自腿部构型和 nominal gait 先验冲突。
> - `track_height_exp(std=0.07) + base_height_error_l1` 证明 height reward 已经开始提供有效信号，但单靠加 height 权重无法突破 nominal-height 局部最优。
> - 最终有效的关键改动是：**standing 只清速度、neutral-only stand still、height L1、低 height 关闭 legs 默认位惩罚、降低 AMP coeff**。这组改动后 height 跟踪开始出现明显效果。
> - gait-free 修改的预期收益是改善 yaw、y-only、低速 command 的运动响应，同时避免把 yaw-mix 专项 reward 全量引入 move operate。它不是“更强的 yaw-only 专项修补”，而是把步态先验从显式 clock 转成更通用的接触/腾空/周期统计约束。
> - 最新判断是：neutral stand 已经基本解决；in-place RPH 要靠足端 anchor 定义“原地姿态控制”；locomotion 还要补侧移几何、touchdown softness 和 event gait 周期，不能只看总 reward。
>
> **一句话总结**
> move operate 的核心不是简单把 command 从 3D 扩成 6D，而是要重新定义“standing”“默认姿态”“原地姿态控制”和“方向相关步态几何”在 6D command 下的语义；尤其 height command 会直接挑战 nominal-height gait，因此必须同时处理 command 语义、standing reward、height reward 形状、腿部默认位约束、AMP 先验，以及 zero-velocity 下 neutral stand / in-place pose 的区别。

在谈关于 gait free 训练范式对比之前依赖 gait 的训练范式，之前的范式有三个问题：**机器人踩地很重；zero command 原地晃悠；yaw/x command 跟踪不理想**。
**核心原因**
旧的 gait 范式把“走路”当成默认吸引子，而不是让 policy 从 command 本身决定是否该走。
在旧 gait 奖励里：
- `gait_feet_frc_perio` 只要求 swing 相脚接触力小。
- `gait_feet_frc_support_perio` 在 stance 相奖励“有接触力”，但 `1 - exp(-100 * force^2)` 很快饱和，所以它不区分“轻柔承重”和“重重砸地”。
- `gait_feet_spd_perio` 要求支撑脚速度小，所以最容易学成“砸下去，然后粘住”。

所以旧范式天然容易 reward hacking：为了确保 stance 相拿分，policy 会选择更确定的强接触，而不是轻落脚。只要没有强的 impact / vertical force magnitude / landing smoothness 惩罚，重踩地就是更稳的局部最优。

**zero command 原地晃悠**
zero command 下晃悠，本质是 “站立目标”和“走路先验” 在打架。
旧范式里有几个走路吸引子：
- gait phase / gait cycle 给 policy 一个持续振荡时钟。
- 旧 gait reward 即使被 `non_zero_flag` gate 掉，policy 也已经在大部分 moving env 中学到了 phase-conditioned 行为，zero command 只是用较弱的 stand-still 奖励去压住这个振荡。
- `stand_still_exp` 这类项主要约束关节回默认位或速度小，不等价于“强制双脚稳定、base 不晃、动作输出归零”。

所以 zero command 下它不是从零开始学“站稳”，而是在一个已经内化的步态振荡器上强行刹车。刹不干净就表现为原地晃、踏步、小跳。

**为什么 gait-free 是对的方向**
旧 gait 先验解决的是“有周期”，但没有解决“轻落脚、按 command 走、zero command 稳定站”。它把训练简化了，但副作用是把接触时序硬塞进 policy。

现在改成 `gait_cycle_reward_event`、`feet_air_time_positive_biped`、`swing_clearance_l1`、`contact_no_slip` 这类 gait-free/event-based 奖励，方向是对的：让接触事件、离地高度、滑移、周期从真实接触状态里出来，而不是让固定 gait phase 直接驱动腿。缺点是训练更难，所以才需要 RPH curriculum 和更强的 standing 稳定项来先把基础步态学稳。

但是之前的范式训练很快；现在的 gait free 范式训练很慢，难以训练，训练波动大。
