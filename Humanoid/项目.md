## 0.基础知识
$\quad\quad$Deploy_Tienkung 是 **ROS2 + C++ 为主**，里面的 rl_control_new、x_humanoid_rl_sdk 是 ROS2 包，里面有 C++ 代码和插件，所以要先 colcon build，它**编译你的包**，然后把结果放到：install/ 目录下。
编译完还要 source ~/tklab_ws/install/setup.bash，这样 ROS2 才知道这些包和可执行节点在哪里。
- ros2 run：跑一个 ROS2 节点
- ros2 launch：按启动文件跑一组 ROS2 节点

> 可以把 PPO 训练想成这个流程：
> 
> 1. 跑环境，采一大批 rollout 数据
> 2. 用这批数据更新策略
> 3. 进入下一轮
> 
> 这里对应关系就是：
> 
> - max_iterations 控制第 1-3 这个大循环跑多少次
> - num_learning_epochs 控制第 2 步里，这批 rollout 要被重复训练多少遍
> - num_mini_batches 控制第 2 步里，每一遍训练把 rollout 切成几块
> - num_steps_per_env控制第1步里面每轮 rollout 时，每个 env 采多少步。
## 1.TienKung-Lab/legged_lab

**如果要设计一个RSL-RL，怎么设计？**
1. 从环境拿 observation
2. 用 policy 网络输出 action
3. 把 action 喂回环境
4. 收集一段 rollout
5. 用 PPO/AMP-PPO 更新网络参数
6. 重复上面过程
先有RSL-RL Runner在有BaseEnv环境，这样顺着看，也就知道env需要怎么写了：给obs，做action。

**如果要设计一个BaseEnv环境，怎么设计？**
- 仿真层
- 实体层
- 状态层
- 指令层
- 控制循环层

**如果要设计一个deploy pipeline，怎么设计？**
1. GetState(...) 读当前机器人状态
2. RunFSM(...) 决定现在该停、归零，还是跑策略
3. SetCommand(...) 把目标命令下发到底层
robot_interface可以拿到机器人传感器的状态给policy，也可以拿到policy的action下发命令给机器人。
具体细节如下：
**先建立一个最小心智模型**

部署本质上不是“一个大 C++ 项目”，而是“一个 400Hz 实时控制循环，加几个功能模块”。
最核心的入口在 main.cpp。每个周期只干 3 步：

1. `GetState(...)` 读当前机器人状态
2. `RunFSM(...)` 决定现在该停、归零，还是跑策略
3. `SetCommand(...)` 把目标命令下发到底层

**这仓库里真正重要的 4 层**

1. 数据面：`RobotData`
在 RobotInterface.h。
它就是整个系统共享的“黑板”。
- `q_a_ / q_dot_a_ / tau_a_`：实际状态
- `q_d_ / q_dot_d_ / tau_d_`：期望命令
- `joint_kp_p_ / joint_kd_p_`：底层并联侧增益
- `velEst / gait_a / pos_mode_`：额外控制状态

你可以把它理解成：所有模块都不直接互相调用业务细节，只通过这份状态结构交换信息。

2. 适配层：`RobotInterfaceImpl`
在 RobotInterfaceImpl.cpp。
它解决的是“机器人世界”和“策略世界”之间的翻译问题。

`GetState()` 做三件事：
- 从 Webots 或实机读关节、IMU 等原始数据
- 如果是实机，先做脚踝串并联到等效串联的转换
- 调状态估计，补出浮基姿态和速度

`SetCommand()` 做反方向三件事：
- 先用 PD 把目标位置变成力矩/执行器参把考
- 如果是实机，再把等效串联命令转回并联机构命令
- 最后交给仿真或底层驱动

3. 决策层：`RobotFSM`
在 RobotFSMImpl.cpp。
这层只管接口。
状态很少：
- `STOP`
- `ZERO`
- `MLP`

这很重要。现实部署里，RL 只是一个状态，不是整个系统。  
机器人不能上电就直接跑策略，必须先经过安全状态和过渡状态。

4. 策略层：`StateMLP`
在 FSMStateImpl.cpp。
这层才是“把 RL 模型接进来”。

它干的事其实非常标准：
- 读 joystick / navigation 指令
- 做 command ramp，避免指令突变
- 手工拼 obs
- 每 8 个控制周期跑一次 OpenVINO 推理
- 把网络输出重排、缩放、加上默认关节位
- 写回 `robot_data_->q_d_`

**现在最该理解的两个“部署难点”**

1. 为什么要有状态估计
训练环境里，base vel、姿态、接触这些量通常直接可得。  
实机没有这么干净，所以要在 StateEstimate.cpp 里自己估。

这里的逻辑是：
- 用 RBDL 建一个简化刚体模型
- 用动力学和关节力矩反推地面反力：StateEstimate.cpp
- 用接触状态给左右脚不同 trust：StateEstimate.cpp
- 再用卡尔曼滤波融合 IMU 加速度和足端运动学，估计腰部速度/高度：StateEstimate.cpp

从第一性原理讲，这层是在回答一个问题：
“机器人现在到底处在什么状态，才能把它喂给策略？”

2. 为什么要有 `P2S / S2P`
你训练里的动作空间通常假设关节是理想串联铰链。  
但这个机器人脚踝那块不是这么简单，所以实机部署前后都要做机构坐标变换。

这块在 funcSPTrans.cpp。
它做的不是“学习”，而是“机械结构补偿”：
- 实际编码器读数先转成策略理解的等效关节状态
- 策略输出的等效关节命令再转回实际执行器命令

如果不分清这一层，你会误以为“策略输出怎么和电机指令不一样”。其实中间多了一层机构映射。

**怎么读这个仓库，效率最高**

建议顺序：

1. 先读 main.cpp
只看主循环，建立闭环概念。

2. 再读 RobotInterface.h
先搞清 `RobotData` 每个字段代表什么。

3. 再读 RobotFSMImpl.cpp 和 FSMState.h
先接受“RL 只是 MLP 状态”这个事实。

4. 再读 FSMStateImpl.cpp
重点看 obs 是怎么拼的，action 是怎么落地的。  
这是和你训练最直接对应的一层。

5. 最后读 RobotInterfaceImpl.cpp、StateEstimate.cpp、funcSPTrans.cpp
这三块是“实机世界”的脏活，不是先读最划算，但迟早得懂。

**最后一句压缩版定义**

这个仓库的核心不是 C++ 技巧，而是这条链：

`传感器/编码器 -> 状态估计 -> obs 拼接 -> 策略推理 -> 关节目标 -> 机构变换/PD -> 电机`

## 2.分布式训练
布式训练里，你可以先把它理解成“每张卡起一个独立进程，各自跑一份环境和一份网络，然后周期性同步”。

这里有两个最基本的同步动作：

1. 启动同步  
    每个 rank 一开始的模型参数必须一致。常见做法是：
    - 要么所有 rank 用同一个随机种子，各自初始化出完全一样的权重
    - 要么只让 rank0 初始化，然后把参数广播给别的 rank
2. 梯度同步  
    每个 rank 各自算完 loss 和 backward 之后，要把梯度做一次平均，也就是 all_reduce，这样每张卡上的 optimizer.step() 才**等价于“一个更大的 batch”在训练**。
    
    梯度同步本质上就是：
- 每个 rank 都有一份 parameter.grad
- 对每个 grad 做 all_reduce(sum)
- 再除以 world_size
- 然后每个 rank 再各自 optimizer.step()

## 3.ROS

先看**没有 ROS** 时，程序怎么工作。

你写一个 C++ 控制程序，比如 main.cpp：

```cpp
int main() {
  while (true) {
    GetState();
    RunFSM();
    SetCommand();
  }
}
```

这件事本质上只是：

1. 操作系统启动一个进程
2. 进程从 `main()` 开始跑
3. 代码自己决定读什么、算什么、写什么

如果你还想接手柄、相机、导航命令，有两种办法。

第一种，不用 ROS。  
那你就自己处理一切：

- 手柄：自己读 `/dev/input/js0`
- 相机：自己开驱动
- 导航命令：自己开 socket / shared memory / file / 串口
- 多个程序之间通信：自己定义协议

这就是“裸进程 + 自己管通信”。timer += dt_
timer_gait += dt_
policy 调用判断
步态相位推进
zero 动作插值

ROS 干的事情，本质上就是把这部分公共问题统一掉。  
所以我说它“多出来的是一层进程组织和通信框架”。

**什么叫进程组织**

假设你现在不是一个程序，而是 5 个程序：

- 一个读手柄
- 一个做状态估计
- 一个做步态控制
- 一个做可视化
- 一个记录日志

不用 ROS 时，你要自己回答：

- 先启动哪个？
- 哪个挂了怎么办？
- 它们之间怎么找到彼此？
- 配置参数放哪？
- 一次怎么把它们全启动起来？

ROS 就是把这些问题抽象了：

- 每个程序叫 `node`
- 节点之间可以被统一启动
- 可以统一传参数
- 可以统一命名
- 可以统一管理谁和谁连接

这就叫“进程组织”。

比如：
- `ros2 run`：启动一个节点
- `ros2 launch`：按一套配置启动一组节点

注意，ROS 没有改变你的控制算法本身。  
它只是帮你“组织多个进程怎么一起工作”。

**什么叫通信框架**

再看节点之间怎么交换数据。

不用 ROS 时，如果手柄程序要把速度命令给控制程序，你得自己设计：

- 数据格式
- 谁发谁收
- 用 TCP、UDP、共享内存还是管道
- 怎么处理频率、丢包、时间戳

ROS 直接给你一套现成模型：

- `topic`：连续数据流  
  例如 `/cmd_vel`、`/joy`
- `service`：请求-响应  
  例如“切换模式”
- `action`：长任务反馈
- `parameter`：运行参数

所以如果有个 `joy_node`，它做的事情其实很朴素：

- 读取 `/dev/input/js0`
- 把结果包装成 ROS 消息
- 发布到 `/joy`

控制节点不用再自己读设备了，只要订阅 `/joy` 就行。

这就是“通信框架”。

**为什么说 ROS 只是多了一层，而不是替代 C++**

因为你的算法代码还在那里。

比如你这个仓库里真正干活的还是：

- main.cpp
- RobotInterfaceImpl.cpp
- RobotFSMImpl.cpp
- FSMStateImpl.cpp

这些是“业务逻辑”：
- 怎么读状态
- 怎么拼 obs
- 怎么跑 policy
- 怎么发控制命令

ROS 不会替你写这些。  
ROS 只是让这些程序更容易和别的程序协作。

所以更准确地说：

- **C++/Python**：解决“程序内部怎么算”
- **ROS**：解决“多个程序怎么连起来一起跑”

**用一个很具体的类比**

把机器人系统想成一个公司。

没有 ROS：
- 每个人自己打电话、发微信、拉群、约时间
- 也能干活，但通信和协作全靠自己维护

有 ROS：
- 公司提供统一办公系统
- 大家有统一通讯录、会议室、流程、公告板

但真正做设计、做代码、做实验的人还是原来那些人。  
办公系统没有替他们工作。

ROS 对机器人程序就是这个“统一办公系统”。

**落到当前这个项目**

你现在 Webots 这条链，基本没怎么用 ROS。

因为是：

- Webots 直接启动 controller
- controller 从 main.cpp 进入
- 然后自己跑控制循环

这是一种“单主程序闭环”。

而如果你启用 `USE_ROS_JOY`，事情就变成：

- 有 ROS 节点负责发布 joystick / navigation 命令
- 你的控制程序去订阅这些消息
- 这时 ROS 开始参与“进程组织和通信”

但即使这时，控制逻辑本体仍然是你的 C++ 代码，不是 ROS 替你控制机器人。

**最短总结**

“ROS 多出来的只是一层进程组织和通信框架”的意思就是：

- 它不替你写控制算法
- 它不替你决定 policy 怎么算
- 它主要负责：
  - 多个程序怎么启动
  - 多个程序怎么互相发消息
  - 参数、命名、服务怎么统一管理

如果你愿意，我下一条可以直接用你这个仓库里的两条实际路径画给你看：

1. `Webots 模式` 到底是谁启动谁  
2. `USE_ROS_JOY 模式` 到底 ROS 多插进了哪一层

## 4.时基
**basicTimeStep**：Webots 物理引擎最小积分步长。
**TIME_STEP**： controller 调 Webots 的周期。代码里真正推进 Webots 的地方是 RobotInterfaceImpl.cpp ：humanoid_.robot->step(TIME_STEP);
- main.cpp (line 174) 的 period(0, 2500000)：主循环墙钟 sleep 周期，2.5ms。这个主要限制程序现实时间节奏，不是 policy 的直接触发器。
- main.cpp (line 178) 的 dt=0.0025：传给状态估计和日志的控制周期假设。
- FSMState.h (line 32) 的 dt_=0.0025：FSM 内部 timer 增量。
- FSMState.h (line 33) 的 freq_ratio_=8：policy 每多少个 FSM tick 推理一次。
- FSMStateImpl.cpp (line 171) 的 LowPassFilter(..., 0.0025, ...)：角速度滤波器假设采样周期是 2.5ms。
- FSMStateImpl.cpp (line 178) 的 gait_cycle=0.85：步态相位周期，不是控制频率，但相位更新依赖 timer_gait += dt_。
- walk_cfg.py (line 433) 的 dt=0.0025, decimation=4：训练侧物理步长和 policy 周期。
```c++
timer = 0.;

if ((int)(timer / dt_) % freq_ratio_ == 0) {
  infer_request.infer();
}

timer += dt_;
```

所以要改源码，统一这些写法。
