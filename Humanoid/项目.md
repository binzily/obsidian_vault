> 可以把 PPO 训练想成这个流程：
> 
> 1. 跑环境，采一大批 rollout 数据
> 2. 用这批数据更新策略
> 3. 进入下一轮
> 
> 这里对应关系就是：
> 
> - max_iterations 控制第 1-3 这个大循环跑多少次

## 1.TienKung-Lab/legged_lab

**如果要设计一个RSL-RL，怎么设计？**
1. 从环境拿 observation
2. 用 policy 网络输出 action
3. 把 action 喂回环境
4. 收集一段 rollout
5. 用 PPO/AMP-PPO 更新网络参数
6. 重复上面过程
先有RSL-RL Runner在有BaseEnv环境，这样顺着看，也就知道env需要怎么写了：给obs，做action。

**如果要设计一个BaseEnv环境，怎么设计？**
- 仿真层
- 实体层
- 状态层
- 指令层
- 控制循环层

**如果要设计一个deploy pipeline，怎么设计？**
1. GetState(...) 读当前机器人状态
2. RunFSM(...) 决定现在该停、归零，还是跑策略
3. SetCommand(...) 把目标命令下发到底层
robot_interface可以拿到机器人传感器的状态给policy，也可以拿到policy的action下发命令给机器人。
具体细节如下：
**先建立一个最小心智模型**

部署本质上不是“一个大 C++ 项目”，而是“一个 400Hz 实时控制循环，加几个功能模块”。
最核心的入口在 main.cpp。每个周期只干 3 步：

1. `GetState(...)` 读当前机器人状态
2. `RunFSM(...)` 决定现在该停、归零，还是跑策略
3. `SetCommand(...)` 把目标命令下发到底层

**这仓库里真正重要的 4 层**

1. 数据面：`RobotData`
在 RobotInterface.h。
它就是整个系统共享的“黑板”。
- `q_a_ / q_dot_a_ / tau_a_`：实际状态
- `q_d_ / q_dot_d_ / tau_d_`：期望命令
- `joint_kp_p_ / joint_kd_p_`：底层并联侧增益
- `velEst / gait_a / pos_mode_`：额外控制状态

你可以把它理解成：所有模块都不直接互相调用业务细节，只通过这份状态结构交换信息。

2. 适配层：`RobotInterfaceImpl`
在 RobotInterfaceImpl.cpp。
它解决的是“机器人世界”和“策略世界”之间的翻译问题。

`GetState()` 做三件事：
- 从 Webots 或实机读关节、IMU 等原始数据
- 如果是实机，先做脚踝串并联到等效串联的转换
- 调状态估计，补出浮基姿态和速度

`SetCommand()` 做反方向三件事：
- 先用 PD 把目标位置变成力矩/执行器参考
- 如果是实机，再把等效串联命令转回并联机构命令
- 最后交给仿真或底层驱动

3. 决策层：`RobotFSM`
在 RobotFSMImpl.cpp。
这层只管接口。
状态很少：
- `STOP`
- `ZERO`
- `MLP`

这很重要。现实部署里，RL 只是一个状态，不是整个系统。  
机器人不能上电就直接跑策略，必须先经过安全状态和过渡状态。

4. 策略层：`StateMLP`
在 FSMStateImpl.cpp。
这层才是“把 RL 模型接进来”。

它干的事其实非常标准：
- 读 joystick / navigation 指令
- 做 command ramp，避免指令突变
- 手工拼 obs
- 每 8 个控制周期跑一次 OpenVINO 推理
- 把网络输出重排、缩放、加上默认关节位
- 写回 `robot_data_->q_d_`

**现在最该理解的两个“部署难点”**

1. 为什么要有状态估计
训练环境里，base vel、姿态、接触这些量通常直接可得。  
实机没有这么干净，所以要在 StateEstimate.cpp 里自己估。

这里的逻辑是：
- 用 RBDL 建一个简化刚体模型
- 用动力学和关节力矩反推地面反力：StateEstimate.cpp
- 用接触状态给左右脚不同 trust：StateEstimate.cpp
- 再用卡尔曼滤波融合 IMU 加速度和足端运动学，估计腰部速度/高度：StateEstimate.cpp

从第一性原理讲，这层是在回答一个问题：
“机器人现在到底处在什么状态，才能把它喂给策略？”

2. 为什么要有 `P2S / S2P`
你训练里的动作空间通常假设关节是理想串联铰链。  
但这个机器人脚踝那块不是这么简单，所以实机部署前后都要做机构坐标变换。

这块在 funcSPTrans.cpp。
它做的不是“学习”，而是“机械结构补偿”：
- 实际编码器读数先转成策略理解的等效关节状态
- 策略输出的等效关节命令再转回实际执行器命令

如果不分清这一层，你会误以为“策略输出怎么和电机指令不一样”。其实中间多了一层机构映射。

**怎么读这个仓库，效率最高**

建议顺序：

1. 先读 main.cpp
只看主循环，建立闭环概念。

2. 再读 RobotInterface.h
先搞清 `RobotData` 每个字段代表什么。

3. 再读 RobotFSMImpl.cpp 和 FSMState.h
先接受“RL 只是 MLP 状态”这个事实。

4. 再读 FSMStateImpl.cpp
重点看 obs 是怎么拼的，action 是怎么落地的。  
这是和你训练最直接对应的一层。

5. 最后读 RobotInterfaceImpl.cpp、StateEstimate.cpp、funcSPTrans.cpp
这三块是“实机世界”的脏活，不是先读最划算，但迟早得懂。

**最后一句压缩版定义**

这个仓库的核心不是 C++ 技巧，而是这条链：

`传感器/编码器 -> 状态估计 -> obs 拼接 -> 策略推理 -> 关节目标 -> 机构变换/PD -> 电机`

## 2.分布式训练
布式训练里，你可以先把它理解成“每张卡起一个独立进程，各自跑一份环境和一份网络，然后周期性同步”。

这里有两个最基本的同步动作：

1. 启动同步  
    每个 rank 一开始的模型参数必须一致。常见做法是：
    - 要么所有 rank 用同一个随机种子，各自初始化出完全一样的权重
    - 要么只让 rank0 初始化，然后把参数广播给别的 rank
2. 梯度同步  
    每个 rank 各自算完 loss 和 backward 之后，要把梯度做一次平均，也就是 all_reduce，这样每张卡上的 optimizer.step() 才**等价于“一个更大的 batch”在训练**。
    
    梯度同步本质上就是：
- 每个 rank 都有一份 parameter.grad
- 对每个 grad 做 all_reduce(sum)
- 再除以 world_size
- 然后每个 rank 再各自 optimizer.step()

## 3.ROS

先看**没有 ROS** 时，程序怎么工作。

你写一个 C++ 控制程序，比如 main.cpp：

```cpp
int main() {
  while (true) {
    GetState();
    RunFSM();
    SetCommand();
  }
}
```

这件事本质上只是：

1. 操作系统启动一个进程
2. 进程从 `main()` 开始跑
3. 代码自己决定读什么、算什么、写什么

如果你还想接手柄、相机、导航命令，有两种办法。

第一种，不用 ROS。  
那你就自己处理一切：

- 手柄：自己读 `/dev/input/js0`
- 相机：自己开驱动
- 导航命令：自己开 socket / shared memory / file / 串口
- 多个程序之间通信：自己定义协议

这就是“裸进程 + 自己管通信”。timer += dt_
timer_gait += dt_
policy 调用判断
步态相位推进
zero 动作插值

ROS 干的事情，本质上就是把这部分公共问题统一掉。  
所以我说它“多出来的是一层进程组织和通信框架”。

**什么叫进程组织**

假设你现在不是一个程序，而是 5 个程序：

- 一个读手柄
- 一个做状态估计
- 一个做步态控制
- 一个做可视化
- 一个记录日志

不用 ROS 时，你要自己回答：

- 先启动哪个？
- 哪个挂了怎么办？
- 它们之间怎么找到彼此？
- 配置参数放哪？
- 一次怎么把它们全启动起来？

ROS 就是把这些问题抽象了：

- 每个程序叫 `node`
- 节点之间可以被统一启动
- 可以统一传参数
- 可以统一命名
- 可以统一管理谁和谁连接

这就叫“进程组织”。

比如：
- `ros2 run`：启动一个节点
- `ros2 launch`：按一套配置启动一组节点

注意，ROS 没有改变你的控制算法本身。  
它只是帮你“组织多个进程怎么一起工作”。

**什么叫通信框架**

再看节点之间怎么交换数据。

不用 ROS 时，如果手柄程序要把速度命令给控制程序，你得自己设计：

- 数据格式
- 谁发谁收
- 用 TCP、UDP、共享内存还是管道
- 怎么处理频率、丢包、时间戳

ROS 直接给你一套现成模型：

- `topic`：连续数据流  
  例如 `/cmd_vel`、`/joy`
- `service`：请求-响应  
  例如“切换模式”
- `action`：长任务反馈
- `parameter`：运行参数

所以如果有个 `joy_node`，它做的事情其实很朴素：

- 读取 `/dev/input/js0`
- 把结果包装成 ROS 消息
- 发布到 `/joy`

控制节点不用再自己读设备了，只要订阅 `/joy` 就行。

这就是“通信框架”。

**为什么说 ROS 只是多了一层，而不是替代 C++**

因为你的算法代码还在那里。

比如你这个仓库里真正干活的还是：

- main.cpp
- RobotInterfaceImpl.cpp
- RobotFSMImpl.cpp
- FSMStateImpl.cpp

这些是“业务逻辑”：
- 怎么读状态
- 怎么拼 obs
- 怎么跑 policy
- 怎么发控制命令

ROS 不会替你写这些。  
ROS 只是让这些程序更容易和别的程序协作。

所以更准确地说：

- **C++/Python**：解决“程序内部怎么算”
- **ROS**：解决“多个程序怎么连起来一起跑”

**用一个很具体的类比**

把机器人系统想成一个公司。

没有 ROS：
- 每个人自己打电话、发微信、拉群、约时间
- 也能干活，但通信和协作全靠自己维护

有 ROS：
- 公司提供统一办公系统
- 大家有统一通讯录、会议室、流程、公告板

但真正做设计、做代码、做实验的人还是原来那些人。  
办公系统没有替他们工作。

ROS 对机器人程序就是这个“统一办公系统”。

**落到当前这个项目**

你现在 Webots 这条链，基本没怎么用 ROS。

因为是：

- Webots 直接启动 controller
- controller 从 main.cpp 进入
- 然后自己跑控制循环

这是一种“单主程序闭环”。

而如果你启用 `USE_ROS_JOY`，事情就变成：

- 有 ROS 节点负责发布 joystick / navigation 命令
- 你的控制程序去订阅这些消息
- 这时 ROS 开始参与“进程组织和通信”

但即使这时，控制逻辑本体仍然是你的 C++ 代码，不是 ROS 替你控制机器人。

**最短总结**

“ROS 多出来的只是一层进程组织和通信框架”的意思就是：

- 它不替你写控制算法
- 它不替你决定 policy 怎么算
- 它主要负责：
  - 多个程序怎么启动
  - 多个程序怎么互相发消息
  - 参数、命名、服务怎么统一管理

如果你愿意，我下一条可以直接用你这个仓库里的两条实际路径画给你看：

1. `Webots 模式` 到底是谁启动谁  
2. `USE_ROS_JOY 模式` 到底 ROS 多插进了哪一层

## 4.Sim2Real
**第一性原理**
真机部署本质上不是“跑一个 policy”，而是每 2.5ms 做一次闭环控制：

1. 读真机状态：电机编码器、速度、电流、IMU、手柄或导航速度。
2. 坐标/单位转换：把真机 CAN 电机 ID、零位、方向、IMU 欧拉角等转换成训练/策略期望的状态。
3. 运行状态估计：从 IMU 和关节状态得到 base 姿态、速度等 policy observation 需要的信息。
4. 运行 FSM：决定当前是 `STOP`、`ZERO` 还是 `MLP`，以及是否让 RL policy 接管。
5. 推理 policy：把 observation 输入模型，得到目标关节动作。
6. 输出低层控制：把 policy 输出转换为目标关节位置/速度/力矩，再通过 PD 参数发到电机驱动。
7. 安全退出：如果状态异常、急停、禁用关节，就停止发力或退出。

在这个仓库里，这个闭环主线在 RLControlPlugin.cpp。

**代码分层**
RLControlPlugin.hpp 是 ROS2 插件入口。它负责订阅和发布，不直接代表策略本身。

它订阅这些真机输入：

`/leg/status`：腿部电机状态。

`/arm/status`：手臂电机状态。

`/imu/status`：IMU 状态。

`/sbus_data/event`：遥控器输入。

`/hric/robot/cmd_vel`：导航速度输入。

它发布这些控制输出：

`/leg/cmd_ctrl`：腿部电机命令。

`/arm/cmd_ctrl`：手臂电机命令。

`/waist/cmd_pos`：腰部回零命令。

`/hric/motion/status`：运控状态反馈。

真正的策略/FSM接口来自这几个头文件：

RobotInterface.h 定义 `RobotInterface`，它是“状态估计 + policy 输入输出转换”的抽象接口。

RobotFSM.h 定义 `RobotFSM`，它决定当前状态机行为。

FSMState.h 定义状态枚举：`STOP`、`ZERO`、`MLP`。

但这些类的实现不在源码里，而是链接进 libx_humanoid_rl_sdk.so。所以如果要对齐我们之前在 `tg2_lab_sdk` 里改过的 half-walk policy 逻辑，核心实现要么已经被重新编译进这个 `.so`，要么你需要拿到 SDK 源码重新生成 `.so`。

**主循环数据流**
主循环一开始创建 FSM 和接口：

RLControlPlugin.cpp

```cpp
RobotFSM *robot_fsm = get_robot_FSM(robot_data);
RobotInterface *robot_interface = get_robot_interface();
robot_interface->Init();
```

这说明 `rl_control` 自己不 new 具体类，而是从动态库里拿工厂函数。第一性原理上，这是一个“ROS2 外壳调用 SDK 控制核心”的架构。

每个控制周期做这些事：

读 ROS 队列里的最新关节和 IMU。

把 32 个全身电机数组压缩成 20 个控制关节：12 腿 + 4 左臂 + 4 右臂。

用 `zero_pos`、`motor_dir`、`zero_offset` 把真机编码器角度变成控制内部角度。

把结果写入 `robot_data.q_a_ / q_dot_a_ / tau_a_`。

把 IMU 写入 `robot_interface->imu_data_`。

调用 `robot_interface->GetState(timeFSM, robot_data)`。

调用 `robot_fsm->RunFSM(flag_)`。

调用 `robot_interface->SetCommand(robot_data)`。

从 `robot_data.q_d_ / q_dot_d_ / tau_d_` 取出目标命令。

再转换回真机电机坐标，发布 `/leg/cmd_ctrl` 和 `/arm/cmd_ctrl`。

关键点是：`robot_data` 是外壳和 SDK 之间的共享数据合同。真机观测写进去，SDK 读它并生成命令，再写回同一个结构。

**真机最危险的地方**
第一类风险是**坐标和关节顺序**。`bodyIdMap` 定义了全身 32 电机的物理 ID 到数组 index 的映射：bodyIdMap.h。主循环又把 32 维映射到 20 维控制关节。如果这里和训练资产、Webots、SDK policy 顺序不一致，policy 会把“髋关节动作”发给“膝关节”或反号，真机会非常危险。

第二类风险是**零位和方向**。配置文件固定在 `/home/ubuntu/data/param/rl_control_new.txt`，里面读 `zero_pos_0...`、`xsense_data_roll`、`xsense_data_pitch`、速度 offset 等。零位或 IMU offset 错了，policy 看到的机器人姿态就是假的。

第三类风险是**时基**。这里控制周期硬编码为 2.5ms：

RLControlPlugin.cpp

RLControlPlugin.cpp

FSMState.h

外壳每 2.5ms 跑一次，但 policy 可能不是每个周期都推理，`FSMState` 里有 `freq_ratio_ = 4`，这通常意味着 policy 控制周期是 `2.5ms * 4 = 10ms`，对应 100Hz。这个必须和训练中的 `sim.dt`、`decimation` 对齐。

第四类风险是 **PD 参数**。这里真机 PD 是硬编码的：

RLControlPlugin.cpp

policy 输出不是直接电流控制，而是经由目标位置/速度/力矩加 PD 下发。如果训练里的 actuator 模型、stiffness、damping 和这里差很多，sim2real 会变差。

**和之前 Webots 部署的关系**

1. `libx_humanoid_rl_sdk.so` 内部加载的是这次 half-walk 的 `lite_half_walk_policy.xml/bin`，不是旧的 `mwr_tg2_lite_0807.xml/bin`。

## 5.串并联
**问题1**：反解这一步的原因我还是不懂，从串联到并联的映射我倒是懂了，映射出来三个量目标关节角、目标关节角速度、目标转矩这三者地位不是一样的吗，最终不就是要驱动现在的关节角到目标关节角来满足机器人policy动作吗，转矩是多少无所谓吧？
**答**：policy 学到的是“我给一个目标角偏差，PD 会产生某个力矩响应”，**不是**“必须精确追踪这个角度轨迹”。
反解不是为了让“真实电机角度轨迹”和训练里的“等效踝关节角度轨迹”逐点一致，而是为了让真实并联机构对 policy 动作的**力学响应**尽量接近训练/仿真里的响应。

**问题2**：但是我们不是已经求出来目标转矩了吗，为什么不直接用呢，而是还要用关节角？
**答**：反解的目标就是让真机并联踝最终产生的力矩响应，尽量接近训练 / sim2sim 里等效串联踝的力矩响应。
关键问题是：我们虽然已经算出了 `目标转矩 tor_d_p`，但真机底层接口未必允许我们“直接、可靠地让电机输出这个转矩”。当前代码走的是电机 PD 位置接口，而不是纯力矩接口。

底层实际控制大概率是这种形式：

```text
tau_motor =
  Kp_motor * (q_cmd - q_actual)
+ Kd_motor * (qd_cmd - qd_actual)
+ tau_ff
```

也就是说，电机真正输出的力矩不是你上层说一个 `tor_d_p` 就完全等于 `tor_d_p`。它由 `pos / spd / kp / kd / tor` 一起决定，尤其由位置误差和速度误差决定。

如果真机接口是理想力矩控制，那当然可以直接用：

```text
motor.tor = tor_d_p
motor.kp = 0
motor.kd = 0
```

但很多机器人电机控制链路不适合这么做，原因通常是：

1. 底层电机/驱动器主要设计为位置/速度/阻抗接口，力矩项可能只是前馈补偿，不是高带宽闭环力矩控制。
2. 直接力矩控制对通信延迟、摩擦、模型误差、限流、保护策略更敏感，真机上更容易抖、软、失控或者保护。
3. 关节位置 PD 是更稳定的“安全外壳”：即使上层算的力矩不准，电机仍围绕当前位置附近输出受限阻抗，而不是完全开放力矩。
4. 当前 `rl_control` 发布的消息同时带 `pos/spd/tor/kp/kd`，但 SDK 在并联踝这里最后把 `tau_d_` 置零，说明这个部署链路选择的是“把目标力矩编码成位置偏差”，而不是直接发力矩前馈。

代码里能看到这个选择：RobotInterfaceImpl.cpp:160 先用 `tor_d_p` 反解 `q_d_p`：

```cpp
q_d_p(i) = (tor_d_p(i) - Kd_p * (qdot_d_p(i) - qdot_a_p(i))) / Kp_p + q_a_p(i);
```

然后把并联踝的直接力矩命令清零：

```cpp
robot_data.tau_d_.segment(10, 2).setZero();
robot_data.tau_d_.segment(16, 2).setZero();
```

所以最终发给真机的不是：

```text
请直接输出 tor_d_p
```

而是：

```text
请用你的电机 PD，追踪我反解出来的 q_d_p；
这样你内部 PD 算出来的力矩约等于 tor_d_p。
```

为什么还要用关节角？因为当前底层的执行接口就是“位置型阻抗控制”。上层想要一个力矩，但底层擅长根据“位置误差”产生力矩。于是上层把力矩转换成位置误差：

```text
tau = Kp * position_error + Kd * velocity_error
```

反解：

```text
position_error = (tau - Kd * velocity_error) / Kp
```

再得到：

```text
q_cmd = q_actual + position_error
```

这就是“用关节角承载力矩意图”。

一个直观例子：

假设某个并联电机当前角度是 `1.00 rad`，底层 `Kp = 100`，速度项先忽略。上层希望它输出 `10 Nm`。

如果直接用位置控制，你应该发：

```text
q_cmd = 1.00 + 10 / 100 = 1.10 rad
```

电机看到 `0.10 rad` 位置误差，就会打出：

```text
100 * 0.10 = 10 Nm
```

这里 `1.10 rad` 不一定是你真的希望它长期到达的轨迹点，它只是为了让电机此刻产生 `10 Nm` 的“虚拟位置命令”。

所以这不是“目标转矩没用”，恰恰相反：`tor_d_p` 是核心；`q_d_p` 只是把这个目标转矩送进位置 PD 底层的一种实现手段。
