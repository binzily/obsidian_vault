2. **解决了双卡训练的问题定义**

我们最初遇到的是：双卡分布式训练在这台机器上起不来，表现为 `ProcessGroupNCCL` 后续报 `CUDA illegal memory access`。环境是 `Ubuntu 22.04.5 + RTX 5090 x2 + Isaac Sim/IsaacLab + humanoid locomotion`。

**分析和实验**

先在 `tk_lab` 里排查，定位到两类独立问题。

- 第一类是 `symmetry` 路由错配。`lite_walk` 这条线原来把不匹配的 symmetry augmentation 接到了错误实现上，导致观测维度不对。这个问题后来在 [__init__.py](/home/eai/Dev/Personal/wbx/Develop/tk_lab/legged_lab/mdp/__init__.py) 和相关 `walk/run cfg` 路由里修正了。
- 第二类是我们自己这套 distributed 同步链路写得不够稳。原来启动时用 `broadcast_object_list` 去同步 GPU `state_dict`，这条路本身就不对；后面我把它改成了更保守的 tensor-level 同步，并理顺了 seed / rank / device 关系。
做完这两类修复后，双卡还是会炸，但 first-fault 已经后移到更底层的 `omni.physx.fabric.plugin / DirectGpuHelper.cpp`。这说明 Python 侧明显写错的部分已经基本排干净了。

然后我们做了关键对照：不用 `tk_lab`，直接跑官方 IsaacLab。

- 官方 `Isaac-Cartpole-v0` 双卡能过。
- 官方 `Isaac-Velocity-Flat-G1-v0` 双卡能稳定复现同类 `CUDA illegal memory access`。
- 这一步把结论收紧到了：问题不是 `tk_lab` 自定义 env 独有，而是官方 `G1 / humanoid locomotion + dual GPU` 在这台机器上也会中招。

之后又去查了官方 issue，最相关的是 `IsaacLab #4011`，以及更早的一些 multi-GPU / PhysX / fabric 相关 issue。结合本地现象，我们进一步判断：更像是 `NCCL 默认 transport/algo 选择 + 本机拓扑/IOMMU/平台组合` 触发的问题，不是 IsaacLab task 逻辑 bug。

最后做了 workaround 实验。按别人 issue 里的做法加：

- `NCCL_SHM_DISABLE=1`
- `NCCL_IB_DISABLE=1`
- `NCCL_ALGO=Ring`

结果：
- 官方 `Isaac-Velocity-Flat-G1-v0` 双卡短跑通过
- `tk_lab` 的双卡训练也能通过
- 本地对照还表明，最关键的最小 workaround 实际上是 `NCCL_SHM_DISABLE=1`；另外两个更像追加 fallback

**最终结论**

结论分两层。

- 对 `tk_lab` 自己来说，确实存在两处该修的实现问题：`symmetry` 路由错配，以及分布式启动/梯度同步链路不够规范。这两处已经修过。
- 但“双卡最终炸不炸”的主因，不在这两处。更底层的问题是：在这台 `5090 x2` 机器上，官方 IsaacLab 的 humanoid locomotion dual-GPU 也会因为默认 NCCL 通信路径而崩。`NCCL_SHM_DISABLE=1` 能有效绕开，说明问题更接近 NCCL/平台/PhysX 组合，而不是上层训练任务代码。

**最终产出**

产出有两块。

第一块是本地工程修复：
- 修正了 `tk_lab` 里的 symmetry 路由和 distributed 同步实现，相关文件就是上面那几处 `mdp / algorithms / runners / train.py`。

第二块是对外输出：
- 我们在单独的 `IsaacLab-pr` fork 里给官方文档 multi_gpu.rst 增加了 `Troubleshooting NCCL Errors` 小节，记录了这类机器上的 NCCL workaround。
- 这份 docs-only PR 已经发出，内容是不改默认行为，只补官方排障说明。

一句话总结就是：  
我们把“`tk_lab` 自己写错的地方”和“官方/平台层的双卡稳定性问题”分开了；前者已修，后者找到了可靠 workaround，并已经沉淀成一份对外文档 PR。
