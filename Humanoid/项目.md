## 0.基础知识
$\quad\quad$Deploy_Tienkung 是 **ROS2 + C++ 为主**，里面的 rl_control_new、x_humanoid_rl_sdk 是 ROS2 包，里面有 C++ 代码和插件，所以要先 colcon build，它**编译你的包**，然后把结果放到：install/ 目录下。
编译完还要 source ~/tklab_ws/install/setup.bash，这样 ROS2 才知道这些包和可执行节点在哪里。
- ros2 run：跑一个 ROS2 节点
- ros2 launch：按启动文件跑一组 ROS2 节点

> 可以把 PPO 训练想成这个流程：
> 
> 1. 跑环境，采一大批 rollout 数据
> 2. 用这批数据更新策略
> 3. 进入下一轮
> 
> 这里对应关系就是：
> 
> - max_iterations 控制第 1-3 这个大循环跑多少次
> - num_learning_epochs 控制第 2 步里，这批 rollout 要被重复训练多少遍
> - num_mini_batches 控制第 2 步里，每一遍训练把 rollout 切成几块
> - num_steps_per_env控制第1步里面每轮 rollout 时，每个 env 采多少步。
## 1.TienKung-Lab/legged_lab

**如果要设计一个RSL-RL，怎么设计？**
1. 从环境拿 observation
2. 用 policy 网络输出 action
3. 把 action 喂回环境
4. 收集一段 rollout
5. 用 PPO/AMP-PPO 更新网络参数
6. 重复上面过程
先有RSL-RL Runner在有BaseEnv环境，这样顺着看，也就知道env需要怎么写了：给obs，做action。

**如果要设计一个BaseEnv环境，怎么设计？**
- 仿真层
- 实体层
- 状态层
- 指令层
- 控制循环层

**如果要设计一个deploy pipeline，怎么设计？**
1. GetState(...) 读当前机器人状态
2. RunFSM(...) 决定现在该停、归零，还是跑策略
3. SetCommand(...) 把目标命令下发到底层
robot_interface可以拿到机器人传感器的状态给policy，也可以拿到policy的action下发命令给机器人。
具体细节如下：
**先建立一个最小心智模型**

部署本质上不是“一个大 C++ 项目”，而是“一个 400Hz 实时控制循环，加几个功能模块”。
最核心的入口在 main.cpp。每个周期只干 3 步：

1. `GetState(...)` 读当前机器人状态
2. `RunFSM(...)` 决定现在该停、归零，还是跑策略
3. `SetCommand(...)` 把目标命令下发到底层

**这仓库里真正重要的 4 层**

1. 数据面：`RobotData`
在 RobotInterface.h。
它就是整个系统共享的“黑板”。
- `q_a_ / q_dot_a_ / tau_a_`：实际状态
- `q_d_ / q_dot_d_ / tau_d_`：期望命令
- `joint_kp_p_ / joint_kd_p_`：底层并联侧增益
- `velEst / gait_a / pos_mode_`：额外控制状态

你可以把它理解成：所有模块都不直接互相调用业务细节，只通过这份状态结构交换信息。

2. 适配层：`RobotInterfaceImpl`
在 RobotInterfaceImpl.cpp。
它解决的是“机器人世界”和“策略世界”之间的翻译问题。

`GetState()` 做三件事：
- 从 Webots 或实机读关节、IMU 等原始数据
- 如果是实机，先做脚踝串并联到等效串联的转换
- 调状态估计，补出浮基姿态和速度

`SetCommand()` 做反方向三件事：
- 先用 PD 把目标位置变成力矩/执行器参考
- 如果是实机，再把等效串联命令转回并联机构命令
- 最后交给仿真或底层驱动

3. 决策层：`RobotFSM`
在 RobotFSMImpl.cpp。
这层只管接口。
状态很少：
- `STOP`
- `ZERO`
- `MLP`

这很重要。现实部署里，RL 只是一个状态，不是整个系统。  
机器人不能上电就直接跑策略，必须先经过安全状态和过渡状态。

4. 策略层：`StateMLP`
在 FSMStateImpl.cpp。
这层才是“把 RL 模型接进来”。

它干的事其实非常标准：
- 读 joystick / navigation 指令
- 做 command ramp，避免指令突变
- 手工拼 obs
- 每 8 个控制周期跑一次 OpenVINO 推理
- 把网络输出重排、缩放、加上默认关节位
- 写回 `robot_data_->q_d_`

**现在最该理解的两个“部署难点”**

1. 为什么要有状态估计
训练环境里，base vel、姿态、接触这些量通常直接可得。  
实机没有这么干净，所以要在 StateEstimate.cpp 里自己估。

这里的逻辑是：
- 用 RBDL 建一个简化刚体模型
- 用动力学和关节力矩反推地面反力：StateEstimate.cpp
- 用接触状态给左右脚不同 trust：StateEstimate.cpp
- 再用卡尔曼滤波融合 IMU 加速度和足端运动学，估计腰部速度/高度：StateEstimate.cpp

从第一性原理讲，这层是在回答一个问题：
“机器人现在到底处在什么状态，才能把它喂给策略？”

2. 为什么要有 `P2S / S2P`
你训练里的动作空间通常假设关节是理想串联铰链。  
但这个机器人脚踝那块不是这么简单，所以实机部署前后都要做机构坐标变换。

这块在 funcSPTrans.cpp。
它做的不是“学习”，而是“机械结构补偿”：
- 实际编码器读数先转成策略理解的等效关节状态
- 策略输出的等效关节命令再转回实际执行器命令

如果不分清这一层，你会误以为“策略输出怎么和电机指令不一样”。其实中间多了一层机构映射。

**怎么读这个仓库，效率最高**

建议顺序：

1. 先读 main.cpp
只看主循环，建立闭环概念。

2. 再读 RobotInterface.h
先搞清 `RobotData` 每个字段代表什么。

3. 再读 RobotFSMImpl.cpp 和 FSMState.h
先接受“RL 只是 MLP 状态”这个事实。

4. 再读 FSMStateImpl.cpp
重点看 obs 是怎么拼的，action 是怎么落地的。  
这是和你训练最直接对应的一层。

5. 最后读 RobotInterfaceImpl.cpp、StateEstimate.cpp、funcSPTrans.cpp
这三块是“实机世界”的脏活，不是先读最划算，但迟早得懂。

**最后一句压缩版定义**

这个仓库的核心不是 C++ 技巧，而是这条链：

`传感器/编码器 -> 状态估计 -> obs 拼接 -> 策略推理 -> 关节目标 -> 机构变换/PD -> 电机`

## 2.分布式训练
布式训练里，你可以先把它理解成“每张卡起一个独立进程，各自跑一份环境和一份网络，然后周期性同步”。

这里有两个最基本的同步动作：

1. 启动同步  
    每个 rank 一开始的模型参数必须一致。常见做法是：
    - 要么所有 rank 用同一个随机种子，各自初始化出完全一样的权重
    - 要么只让 rank0 初始化，然后把参数广播给别的 rank
2. 梯度同步  
    每个 rank 各自算完 loss 和 backward 之后，要把梯度做一次平均，也就是 all_reduce，这样每张卡上的 optimizer.step() 才**等价于“一个更大的 batch”在训练**。
    
    梯度同步本质上就是：
- 每个 rank 都有一份 parameter.grad
- 对每个 grad 做 all_reduce(sum)
- 再除以 world_size
- 然后每个 rank 再各自 optimizer.step()
