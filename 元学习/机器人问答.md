## 0.机器人本质
我们用**第一性原理**来拆开看这件事：

你看到的流程是：

```
策略 (policy)
↓
控制器集成
↓
机器人通信
↓
真实机器人运行
```

这其实是在解决一个最根本的问题：

> **如何把一个数学函数（policy）变成真实世界机器人的物理动作。**

强化学习最后得到的只是一个**函数**：

[  
a = \pi(s)  
]

意思是：

```
输入：机器人状态 s
输出：控制动作 a
```

但真实机器人不是函数，它是一个**物理系统**。  
所以必须有一整套系统把这个函数**嵌入真实机器人的控制循环**。

下面一步一步用第一性原理拆开。

---

### 一、第一性原理：机器人控制的本质是什么

机器人在物理世界里，本质是：

```
传感 → 计算 → 执行
```

循环：

```
state_t  → controller → action_t
```

也就是：

```
观测
↓
决策
↓
执行
```

这个循环通常是 **200Hz~1000Hz**。

例如：

```
每 5ms
```

系统要完成：

```
读取传感器
运行控制算法
发送电机命令
```

---

### 二、强化学习策略只是“决策函数”

强化学习训练完得到：

```
policy.pt
```

它只是一个神经网络。

本质是：

```
f(x) = y
```

输入：

```
机器人状态
```

输出：

```
关节命令
```

但问题是：

**机器人本身不会运行 PyTorch。**

所以需要把它嵌入机器人控制系统。

---

### 三、控制器集成（Controller Integration）

这是第一步。

目的：

> **把神经网络策略嵌入机器人控制循环**

机器人控制器通常是 C++ 写的。

例如：

```
while(true):

    state = readSensors()

    action = policy(state)

    sendMotorCommand(action)
```

这里的关键是：

#### 1 读取状态

机器人传感器：

```
IMU
joint position
joint velocity
foot contact
```

这些组合成：

```
observation
```

---

#### 2 运行策略

策略通常是：

```
MLP
```

例如：

```
obs → 512 → 256 → action
```

运行一次前向：

```
action = network(obs)
```

---

#### 3 输出控制命令

例如：

```
desired joint position
desired torque
desired velocity
```

然后发给电机控制器。

---

### 四、机器人通信（Robot Communication）

第二步。

机器人系统不是一个单进程程序。

它通常是一个**分布式系统**。

结构类似：

```
计算机
↓
网络
↓
机器人主控
↓
电机驱动
```

所以需要通信。

例如：

```
PC → robot
```

发送：

```
joint commands
```

同时接收：

```
sensor data
```

---

Unitree 使用的是：

```
DDS (Data Distribution Service)
```

这是一种实时机器人通信协议。

类似：

```
ROS2
```

但更底层。

---

通信结构是：

```
policy process
↓
DDS
↓
robot controller
↓
motor drivers
```

---

### 五、运行在真实系统

第三步。

这一步最关键。

仿真里的控制：

```
policy → simulator
```

但真实机器人：

```
policy
↓
controller
↓
communication
↓
robot hardware
↓
physics
```

差异很大。

因为真实系统存在：

```
延迟
噪声
摩擦
模型误差
```

例如：

```
policy计算 3ms
通信延迟 5ms
电机响应 8ms
```

总延迟：

```
16ms
```

而仿真可能是：

```
0ms
```

这就是 **sim2real gap**。

---

### 六、为什么必须有这一整套 pipeline

如果没有：

```
controller integration
communication
deployment
```

那么 RL 代码只能：

```
在仿真跑
```

永远不能：

```
控制机器人
```

所以机器人公司最重要的工程之一就是：

> **把策略嵌入真实控制系统**

---

### 七、用第一性原理总结整个流程

从最底层看：

机器人系统本质是：

```
物理系统
```

你必须解决：

```
如何实时控制物理系统
```

强化学习只解决：

```
如何计算动作
```

但真实机器人需要：

```
如何获取状态
如何运行策略
如何发送命令
如何保证实时性
如何保证安全
```

所以完整系统变成：

```
Sensors
↓
State estimation
↓
RL policy
↓
Controller
↓
Communication
↓
Motor drivers
↓
Physics
```

---

### 八、最核心的一句话

控制器集成 + 机器人通信 + 真实系统运行，本质是在解决：

> **如何把一个神经网络决策函数嵌入到一个实时控制的物理系统中。**

## 1.Actuator 和 Controller 的区别

### 一、Actuator 和 Controller 的区别

先给你一句**最核心的区别**：

> **Controller 决定“应该怎么动”，Actuator 负责“真的让它动起来”。**

---

#### 1 Controller（控制器）

控制器是一个**算法 / 计算模块**。

作用是：

```
输入：当前状态
输出：控制指令
```

例如：

```
机器人状态：
关节角度
关节速度
IMU

↓

控制器计算

↓

关节目标位置
```

例如：

```
policy(state) → action
```

强化学习里的 **policy 本质就是 controller**。

例如：

```
RL policy
MPC
PID controller
```

这些都是 controller。

---

#### 2 Actuator（执行器）

Actuator 是**物理设备或其模型**。

作用：

```
把控制信号 → 变成机械运动
```

例如：

```
控制器输出：

target_position = 0.5

↓

电机驱动

↓

关节真的转到 0.5
```

所以 actuator 是：

```
电机
减速器
驱动器
```

组合。

---

#### 3 控制系统的完整结构

机器人系统其实是：

```
Controller
    ↓
Control Signal
    ↓
Actuator
    ↓
Robot Dynamics
```

例如：

```
RL policy
↓
target joint position
↓
motor
↓
joint movement
```

---

#### 4 一个简单类比（人体）

人体结构：

```
大脑 = controller
神经信号 = control signal
肌肉 = actuator
身体运动 = robot dynamics
```

大脑决定：

```
抬手
```

肌肉负责：

```
真的抬起来
```

---

### 二、为什么 RL locomotion 项目会有两种 actuator

很多 locomotion 项目里会看到：

```
ImplicitActuator
IdealPDActuator
```

原因是：

> **仿真中的 actuator 和真实电机不一样**

这就会产生：

```
Sim2Real gap
```

所以研究者会设计不同的 actuator 模型。

---

### 三、第一种 actuator：ImplicitActuator

特点：

```
非常稳定
计算简单
训练快
```

它基本等价于：

```
直接控制关节
```

仿真器内部帮你完成控制。

例如：

```
action → joint torque
```

PhysX / Isaac Sim 会直接执行。

优点：

```
训练稳定
收敛快
```

缺点：

```
不真实
```

因为真实电机不是这样工作的。

---

### 四、第二种 actuator：IdealPDActuator

这个更接近真实电机。

控制公式：

```
torque = Kp*(target_pos - pos)
       + Kd*(target_vel - vel)
```

流程是：

```
policy
↓
target position
↓
PD controller
↓
motor torque
↓
robot motion
```

这种模型更接近真实机器人。

优点：

```
更真实
更接近真实控制器
```

缺点：

```
训练更难
容易不稳定
```

---

### 五、为什么 RL 项目会同时提供两种

因为机器人训练通常有两个阶段。

---

#### 阶段1：训练稳定性

用：

```
ImplicitActuator
```

因为：

```
收敛更快
更稳定
```

先让策略学会：

```
走路
```

---

#### 阶段2：Sim2Real

换成：

```
IdealPDActuator
```

让策略适应：

```
真实电机动力学
```

这样更容易：

```
部署到真实机器人
```

---

### 六、为什么 actuator mismatch 会导致机器人摔倒

如果训练用：

```
ImplicitActuator
```

真实机器人用：

```
PD actuator
```

差异：

```
延迟
带宽
扭矩限制
```

RL 策略就会：

```
控制过激
不稳定
```

机器人就会：

```
摔倒
```

这就是很多 locomotion 论文研究的问题：

```
actuator modeling
```

---

### 七、机器人 RL 的完整控制结构

很多 locomotion 系统实际是：

```
RL policy (controller)
↓
target joint position
↓
PD controller
↓
motor torque
↓
robot dynamics
```

所以 RL policy **不是直接控制电机**。

而是：

```
控制 PD controller 的目标
```

---

### 八、一句话总结

### actuator vs controller

```
controller = 算法
actuator = 电机
```

### 两种 actuator 的原因

```
ImplicitActuator → 训练稳定
IdealPDActuator → 更真实
```
