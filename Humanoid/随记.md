2. **解决了双卡训练的问题定义**
**核心结论**

这次双卡 `illegal memory access` 的问题，最终没有落到 IsaacLab 的 task 逻辑本身。更准确地说：

- IsaacLab 这一层只是启动 distributed training。
- `rsl_rl` 这一层也只是做 `torch.distributed` / `init_process_group("nccl")` 之类的初始化。
- 真正决定通信 transport / algo 的，是更底层的 NCCL 默认行为。
- 在我们这台机器和这类 humanoid locomotion workload 上，默认 NCCL 路径会触发崩溃；显式改 `NCCL_SHM_DISABLE=1` 等环境变量后，问题可以绕开。

所以这不是“IsaacLab 某个 task 配错了”，也不是“rsl_rl 里缺了这三个参数的代码开关”这么简单。更准确的说法是：**IsaacLab / rsl_rl 当前没有给用户一个显式的 NCCL 排障入口，而默认 NCCL 选择在部分机器上不稳定。**

**做了什么分析**

先是排除了自己代码里的明显问题。

- 修了 `tk_lab` 里 symmetry 路由错误，避免 `lite_walk` 观测错误地走到不匹配的 symmetry 实现。
- 修了 `tk_lab` 里分布式初始化/同步链路里几处不合理实现，包括 GPU 参数同步方式、seed 处理、梯度同步细节和设备绑定问题。

这些修完以后，双卡错误确实往后推了：
- 不再死在 Python 层的 broadcast / symmetry 维度错配
- first-fault 变成更底层的 `omni.physx.fabric` / `DirectGpuHelper.cpp` / `ProcessGroupNCCL` 一类 CUDA illegal memory access

这一步很重要，因为它说明：`tk_lab` 里原先确实有局部 bug，但它们不是最后那个双卡崩溃的主因。

**我们做了什么实验**

然后用**官方链路做最小复现**，目的是切开“平台问题”和“自定义 env 问题”。

做过的关键对照是：

- 官方 `Isaac-Cartpole-v0` 双卡 distributed：能跑
- 官方 `Isaac-Velocity-Flat-G1-v0` 双卡 distributed：默认会炸
- 你自己的 `tk_lab lite_walk` 双卡 distributed：默认也会炸
- 在github查找相关**Issue**发现加上
  - `NCCL_SHM_DISABLE=1`
  - `NCCL_IB_DISABLE=1`
  - `NCCL_ALGO=Ring`
  之后，官方 G1 和 `tk_lab` 的双卡短跑都能过

更关键的是做了更细的**对照归因**：
- 在这台机器上，`NCCL_SHM_DISABLE=1` 单独就已经足够让官方 `Isaac-Velocity-Flat-G1-v0` 的双卡最小复现通过
- `NCCL_IB_DISABLE=1` 和 `NCCL_ALGO=Ring` 更像附加 fallback，不是第一主因

结合 **NCCL debug** 输出，我们看到：
- 默认失败时，NCCL 实际走的是 `SHM/direct/direct`
- workaround 成功时，通信路径切到了 `NET/Socket/...`

这进一步支持一个判断：**问题更像是默认 NCCL SHM 路径在这台 Linux 双卡 5090 机器上的兼容性/稳定性问题，而不是 IsaacLab 的任务逻辑问题。**

**最终判断**

可以把最终判断压缩成这几句：

- 这个问题不是由 IsaacLab humanoid task 本身触发的逻辑错误。
- 也不是因为 IsaacLab 或 `rsl_rl` 代码里显式把 `NCCL_SHM_DISABLE`、`NCCL_IB_DISABLE`、`NCCL_ALGO` 设错了；实际上它们根本没有显式设置这三个变量。
- IsaacLab / `rsl_rl` 做的事情主要是初始化 distributed training，之后 NCCL 会按默认规则自行选择 transport / algo。
- 在我们本地的 humanoid locomotion 双卡场景里，正是这个默认 NCCL 选择出现了问题。
- 因此，最合适的 upstream 产出不是“硬改 IsaacLab 默认行为”，而是先补一条官方 troubleshooting 文档，告诉用户在受影响机器上如何通过 NCCL 环境变量规避问题。

  - 这是 NCCL 层面的排障说明

## 4.23
1.Webots做sim2sim，在Zero2MLP的时候，还没给手柄命令，机器人**全身抖动**。
分析问题：在 RobotInterfaceImpl::SetCommand 里面，之前的代码大概是这样：
```c++
部分关节:
  setTorque(PD torque)

另一些关节:
  q_d = 500000  // 跳过 setPosition
  tau_d = 500000 // 跳过 setTorque

同时又调用:
  setMotorPos(...)
  setMotorTau(...)

```
这里有几个问题：
1. 500000 不是控制策略，只是一个“别设置这个关节”的哨兵值。结果就是某些关节在 MLP 后可能保持 Webots 上一次 position target，某些关节切到 torque target，某些关节实际等于半开环。
2. Webots 的 setPosition() 是 Webots 内部位置伺服，不是我们训练里的 Isaac actuator PD。它内部还有自己的 motor solver、速度限制、力矩限制。你再同时对别的关节 setTorque()，整个机器人就不是一个统一的控制系统。
3. half-walk 训练时的假设是：12 个腿关节由 policy 给目标位，waist/arms 追默认位，所有关节都通过同一套 PD/隐式 actuator 行为进入物理世界。之前部署侧实际变成：腿、腰、手臂、手腕的控制模式不一致。
4. ZERO2MLP 的瞬间最容易暴露这个问题。ZERO 阶段是 position mode，机器人被拉到默认姿态；一进 MLP，部分关节突然切 torque，部分关节还残留 position servo 或不再更新。这个切换本身就是一个冲击源，所以摇杆不动也会原地抖。

现在改成统一显式 PD torque 后，语义变清楚了：
```c++
21 个训练合同关节：legs + waist + shoulder_pitch/roll/yaw + elbow_pitch
走 torque PD

6 个训练里不存在的额外关节：elbow_yaw / wrist_pitch / wrist_roll，左右各 3 个
只 position hold 默认位

训练过、policy 观测/输出覆盖的关节：可以 torque PD 跟 policy target
没训练过、policy 不知道的额外关节：position hold 默认位，Webots 内部自己处理位置保持，尽量从动力学里“安静掉”

所以手腕这种 policy 没训练过、也不参与主运动的额外关节，用 torque PD 可能会把速度估计噪声、微小位置误差、离散导数误差放大成抖动 torque。尤其 wrist/elbow_yaw 这种轻、小、惯量低的末端关节，kd 或速度噪声稍微不合适就会抖。

```
所以机器人不再被两套控制器撕扯，ZERO 到 MLP 的控制模式切换也一致了，抖动自然消失。
